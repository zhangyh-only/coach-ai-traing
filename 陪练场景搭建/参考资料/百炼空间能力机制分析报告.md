# Coach 陪练 · 百炼空间能力机制分析报告（v1）

> **定位**：这是场景搭建的**架构地基**，优先级高于任何提示词。核心命题——**Coach 的几条红线（角色不真实/话术僵硬、评估不准/建议不能用、价格口径不准、困在单品、对话推进失控），靠"写更好的全量提示词 + 直调大模型"是解决不了的；必须用百炼平台的机制（应用模式选型 / 工作流编排 / 多智能体协作 / RAG 动态检索 / 插件取数）去突破效果上限。**
> **能力来源**：阿里云百炼官方文档（见文末「来源」），逐项核实；标 🔧 为基于能力的架构推断（需控制台实操验证）。
> **关联**：[场景搭建_总体设计与进度.md](../0_项目总览.md) ｜ 上游 [场景搭建_初始输入prompt.md](场景搭建_初始输入prompt.md)

## 0. 为什么这份报告是地基：红线与"全量 prompt"的死结

现状（线上系统 + 标准产品 + 我此前的 Elena 草案）本质都是**一个全量提示词塞进一个大模型直调**。这种形态有三个结构性死结，无论 prompt 写得多好都绕不过：

1. **职责挤压** → 角色僵硬：一个 prompt 既要"演顾客"，又要"评销售好坏"，又要"管阶段推进"，又要"挑异议"。模型每说一句话前先在脑内跑评级/判定流程，输出必然像填表（标准产品 672 行 prompt 就是活证据）。
2. **事实靠记忆** → 评估不准、价格臆造：产品价格、政策、材质事实如果只写在 prompt 里（或靠模型自带知识），就会过期、会幻觉。R4「价格口径准」在纯 prompt 形态下**无解**。
3. **内容写死** → 困在单品、异议重复：产品、异议库全量内联进 prompt，导致顾客被绑死在一只包上（违背 R2）、异议机械复读。

**百炼的价值正在于：把这些职责拆给不同机制承载**——评估归评估流程、事实归 RAG/插件、内容归知识库动态检索、协作归多智能体。这才是"提升效果上限"的真正杠杆。

## 1. 百炼能力全景（官方文档核实）

### 1.1 三种应用模式 + 智能体编排（选型的根）

| 模式 | 定义/能力 | 适用 | 对 Coach 的意义 |
|------|-----------|------|----------------|
| **智能体应用 Agent 2.0** | 提示词驱动，自主理解意图、规划决策，知识库+MCP 统一为工具自主调用；规划-执行-反思 | 开放式对话（客服/问答/助理） | **角色扮演链**主力（顾客需自主、灵活反应） |
| **工作流应用 Workflow** | 可视化节点编排，多步骤串成**稳定可控、可复现**的链路，每步预设节点 | 多步骤、需稳定可控、条件分支、组合多模型/API | **评估两链**主力（评估是"定位→检索维度→打分→出报告"的固定流程，要的就是稳定可复现） |
| **智能体编排 Multi-Agent** | **智能体群组节点**：决策模型自动规划、调度多个子智能体，串行/并行/自动规划；子智能体配"功能描述"供决策模型判断调用 | 复杂多任务、需分工 | **角色扮演链的进阶选项**：把"说话/观察评判/异议调度"拆成协作子 agent，根治僵硬 🔧 |
| 高代码应用 | Python 后端，复杂集成 | 私有算法 | 本期不涉及 |

> **关键纠偏**：初始输入"锁定"的是"三链统一 Agent 2.0"。但按能力匹配，**评估链其实更适合工作流应用**（稳定可控 > 自主发挥），这点见 §5 建议。

### 1.2 工作流节点清单（评估链的积木）
- **控制**：开始/结束、条件判断（且/或分支）、循环、批处理（并发）
- **AI**：大模型（"智慧大脑"）、知识库检索（RAG）、**意图分类**（识别意图选链路）、**参数提取**（从文本抽结构化字段）
- **数据**：脚本（JS/Python）、变量处理、变量赋值
- **外部集成**：API 节点（自定义 REST）、函数计算、插件、MCP、**应用组件节点**（复用已发布的 agent/工作流）
- **流转**：节点间变量传递、本节点缓存 / 全局缓存、流式输出

### 1.3 组件机制（三链互联）
已发布的智能体或工作流，可作为**应用组件节点 / 工具**被其他应用引用。→ 三链可互相调用、也可被业务系统统一编排。

### 1.4 知识库 RAG 精调（事实层锁死的工具）
- **智能切分**（按语义自适应，避免语义截断）
- **召回 TopK**（1–20；总结/列举/比较类问题调高到 20）
- **Rerank 重排序**（排序模型二次精排，提准）
- **元数据 / 标签过滤**（先按标签筛、再向量检索；降 token、提准）—— 对应"单库 + 双标签"
- **相似度阈值**（控相关性下限，命中测试反复调）
- **多路召回 / 混合检索**（向量 + 结构化）
- **调用方式：必定召回 / 按需召回**
- **结构化知识库**（元数据提取提升结构化搜索）

### 1.5 插件 / MCP / API（外部能力接入 = 解 R4 的钥匙）
- **官方插件**：Python 代码解释器、计算器、图片生成、夸克搜索（实时联网）、二维码、GitHub
- **自定义插件**：接 REST API（→ 接 Coach 产品价格/库存/活动价**主数据**）
- **MCP**：官方 + 自定义 MCP 服务，动态调用
- **工作流 API 节点 / 函数计算节点**：直接调企业内部系统

### 1.6 Agent 2.0 关键配置
工具自主规划调用 ｜ 规划-执行-反思 ｜ 知识库标签过滤 ｜ **短期记忆 0–30 轮**（单会话内）｜ **ReAct 最大轮次 1–50** ｜ 模型选千问-Max（强工具调用）｜ 自定义变量（提示词内 `/` 引用）｜ 回复展示来源角标

### 1.7 记忆库 / 长期记忆（跨会话个性化 = 破"每次差不多"的进阶机制）
- **机制**：自动"提取 → 存储 → 检索 → 注入"——对话结束提取关键信息持久化，新对话语义检索召回注入上下文。
- **存两类**：记忆片段（自动提取的关键事件，如"练过价格异议"）+ 用户画像/记忆变量（结构化属性，如学员水平/偏好）。
- **隔离**：按 **user_id / Memory ID** → 可**学员级隔离**（记住"这个 SA 的对练历史"）。
- **接入**：智能体应用可配置长期记忆体并绑定；或 API（AddMemory / SearchMemory）。
- ⚠️ 注：Agent 2.0 内置配置项曾标"长期记忆暂不支持"，但平台有独立**记忆库/长期记忆**能力可接入——以控制台实操为准验证。

## 2. 红线问题 → 机制解法映射（本报告核心）

| 红线 | "全量 prompt + 直调"为何解决不了 | 百炼机制解法 |
|------|----------------------------------|--------------|
| **角色不真实 / 话术僵硬**（R1/R6） | 一个 prompt 既演又评又管阶段，职责挤压 | **拆职责**：角色扮演只管"演"（Agent 2.0）；进阶用**智能体编排**把"说话 / 内部观察评判 / 异议调度"拆成子 agent。**表达层放开口语情绪，评判搬出去** |
| **评估不准 / 建议不能用**（R3） | 评估混在对话里、靠模型自觉、无稳定流程 | 评估用**工作流应用**：定位→**知识库检索 KB5 维度（必定召回）**→逐维度大模型打分→汇总。流程稳定可复现，维度依据来自 RAG 不靠记忆 |
| **价格 / 活动口径不准**（R4） | prompt 里没有实时数据 → 臆造 | **插件 / 自定义 MCP / API 节点**接产品价格/库存/活动**主数据**，实时取数；无数据则返回"待补"，绝不臆造 |
| **困在单品 / 多品类**（R2） | prompt 写死"我要 Tabby" | 产品改为**软背景变量**；用**意图分类 / 参数提取**识别"导购在推荐哪款"，让顾客自然响应多品类；产品 FAQ 沉知识库按款检索 |
| **对话推进失控**（④） | 阶段推进全靠模型在长 prompt 里自觉 | Agent 2.0 **ReAct 轮次 + 自定义变量 + 短期记忆≥20 轮**控阶段；或工作流编排阶段流转 |
| **异议僵硬 / 重复** | 异议库全量内联、固定 | 异议库**沉知识库 RAG**，按"当前阶段 + 导购话语"**按需检索 Top-K**，不全量灌入 → 动态、不重复 |
| **入口/类型可选随机**（R5） | — | 自定义变量（画像可"随机"）+ 业务系统调用时传参 |

## 3. 三链的百炼机制选型与架构

### 3.1 链一 · 角色扮演对话
**目标**：演活顾客（R1/R2/R6 + 对话推进）。

**方案 A（稳健起步）· Agent 2.0**
- 系统提示词只留**轻量人设 + 行为协议 + 阶段心理曲线**（不再内联异议库/产品大段，砍掉评级引擎）。
- **画像细节 / 异议库 / 产品 FAQ → 沉知识库，按需 RAG 检索**（标签：画像类型 / 阶段 / 产品）。异议按"当前阶段 + 导购刚说的话"动态召回 Top-K，**不全量内联** → 解异议僵硬、缩 prompt。
- **自定义变量**注入场景选择项（渠道/画像/阶段/主题/互动模式/难度）；产品为**软背景**（"在看包、对某类有点意向"），不写死单品。
- **短期记忆 ≥20 轮**保人设连贯；ReAct 轮次配合阶段推进。

**方案 B（上限更高）· 智能体编排 🔧**
- 子 agent 拆分：① **顾客说话 agent**（只管自然说话、吃画像变量、表达层放开）② **内部观察 agent**（判断导购行为质量/是否触发情绪变化，**结果不输出给用户**，只influence说话 agent）③ **异议·阶段调度 agent**（吃 KB 异议库 RAG + 阶段状态，决定本轮推进还是抛哪个异议）。
- 价值：把"演"和"判"彻底分离 → 这是根治"边演边评级=僵硬"的架构级解法。代价：编排与调试更复杂。
- **建议**：先用方案 A 打通、验证；若僵硬仍明显，再升方案 B。

### 3.2 链二 · 实时评估
**目标**：即时信号（R3/R4/R7），不出总分。
**机制**：**工作流应用** 🔧
- 开始（取最近 N 轮对话）→ 知识库检索（**KB5 维度判断 + 红线，必定召回**）→ 大模型节点（只评**最新一条销售消息**：本轮异议处理到位？踩红线？给即时观察+改进+口语化参考话术方向）→ 结束（Markdown 即时提示）。
- **事实层锁死**：涉及产品/价格口径，走知识库/插件取数核对，不臆造。

### 3.3 链三 · 整体评估
**目标**：四维度完整评分报告（R3，聚焦"准 + 建议能用"）。
**机制**：**工作流应用** 🔧
- 开始（全程对话）→ 知识库检索（**KB5 四件套，必定召回**）→ [循环/并行] 逐维度大模型打分（**三级评分卡**：维度→指标→评估要点+显式分值组成；**完成度→分数段映射**；**培训场景宽容评分**）→ 大模型汇总（专业性30/同理心25/沟通技巧25/流程意识20，总分=100，三档，证据导向）→ 结束（Markdown 报告，分维度+证据+建议）。
- **事实/表达分离**：事实（产品/价格/政策）用 RAG 锁死；表达（建议/示范话术）放开口语共情 → 解 R3"评价正确但话术不打动"。

### 3.4 跨链与主数据
- **组件机制**：三链各自发布；业务系统按需调用，或用组件节点互相引用。
- **主数据**：自定义插件 / MCP / API 节点接价格·库存·活动价（解 R4 的硬依赖）。

## 4. 关键机制专题

### 4.1 事实层 / 表达层分离（解"评估建议僵硬""价格臆造"）
- **事实层**（产品/价格/材质/政策）：一律走 **RAG 必定召回 + 元数据过滤 / 插件取数**，作为唯一事实源，与检索冲突以检索为准，无数据标"待补"——**锁死，防幻觉**。
- **表达层**（顾客话术 / 评估建议 / 示范话术）：**放开口语、情绪、共情**，不钉死字数句式。
- 这是贯穿三链的总原则（已写入总体设计 §12.6）。

### 4.2 异议库 / 画像 / 产品 FAQ 的 RAG 化（解"困在单品""异议僵硬"）
- 把异议库、画像细节、产品 FAQ **从 prompt 搬进知识库**，用标签（画像/阶段/产品/主题）组织，**按需检索**。
- 产品**软绑定**：场景只设"她在看包、对 X 类有点意向"，导购推荐别的款时，顾客经由检索到的多品类 FAQ **自然响应**，不被拉回单品 → 正面解 R2。

### 4.3 多智能体拆分（解"角色僵硬"的上限解法）
见 §3.1 方案 B。核心：演 / 判 / 调度分离。

### 4.4 主数据接入（解 R4）
价格/库存/活动价用插件或 MCP 实时取；这是初始输入 §9 待确认项之首，**直接决定价格异议场景能否成立**。

### 4.5 破"每次会话差不多"（客户反馈的真问题，经 4 视角对抗评审重做）
**根因**：客户感知的"差不多"是**中段博弈轨迹雷同**（不是开场背景不同）；同一 prompt 下中段对话回归表达均值。
**两个正交轴**（分别配机制）：
- **轴一·跨会话不同（主力）**：开局**随机抽 1 条「过招剧本骨架」**（异议顺序/松口阈值/拉扯/成交路径不同），工作流脚本节点 random 抽 + 最近 N 局不重复。**身份固定 Elena**——变过招路数、不变人/场景（**适度，不过度随机**：4 骨架共享统一底色）。
- **轴二·同会话内动态**：异议 RAG 按导购话检索 + 响应式松口/加压（学员驱动，取之不尽、不重复、不冲突）。
**记忆库（二期）**：按 user_id 做成长追踪 + 按短板偏向抽骨架；**不做去重**（去重用轮转/业务 DB）。
> 详见 [专题_角色多样性与随机机制设计.md](../公共_跨场景复用/随机机制/专题_角色多样性与随机机制设计.md)（v2，含对抗评审加固）。
> ⚠️ v1 曾押"换背景/情绪"与"随机实例化身份"，经评审判为治标不治本 / 与"身份固定"矛盾，**已废止**。

## 5. 与初始输入"锁定方案"的差异与建议（需业务拍板）

初始输入 §1/§9 把平台决策"锁定"为：① 三链统一 Agent 2.0 ② 单库+标签 ③ 全 Markdown ④ 3 个独立 agent、业务系统编排。基于能力分析，提出**三点重估建议**（不擅自推翻，供决策）：

| # | 锁定项 | 报告建议 | 理由 |
|---|--------|----------|------|
| 1 | 三链统一 Agent 2.0 | **评估两链先 Agent 2.0 起步，工作流作升级选项** | 评估可先用 Agent 2.0（必定召回 KB5 + 评分卡 + 固定输出模板），统一、快；**若实测打分不稳/漏维度/事实漂移，再升级工作流**把流程固化（用户质疑"评估为啥工作流"后调整）|
| 2 | 异议/画像写进提示词 | **异议库/画像/产品 FAQ 沉知识库 RAG，按需检索** | 解异议僵硬、困单品；缩 prompt |
| 3 | 角色扮演 Agent 2.0 | 先 A（Agent 2.0）跑通，**预留 B（智能体编排）** | 若僵硬仍明显，用多 agent 拆"演/判" |
| 4 | 单库+标签 ✅ | 维持，符合 RAG 元数据过滤最佳实践 | — |
| 5 | 全 Markdown ✅ | 维持（评估报告用 Markdown 结构化排版） | — |

## 6. 落地路线与待确认

**落地顺序（先角色扮演、再评估，呼应用户优先级）**：
1. 知识库就绪：45 篇 KB 上传 + 双标签；异议/画像/产品 FAQ 可被按需检索。
2. **链一角色扮演（Agent 2.0）**：轻量人设+变量+RAG异议+软产品 → 跑通对话真实度。
3. 链二/三评估（工作流应用）：搭评估流程，事实层 RAG 锁死。
4. 主数据接入（插件/MCP）补 R4。
5. 冒烟 → 功能 → 边界 → 调优。

**待确认（业务/平台）**：
- a. §5 的应用模式重估（评估链改工作流？角色扮演留 Agent 还是上多 agent？）——**这是最关键的架构决策**。
- b. 主数据接入方式（插件 / MCP / API）与可用时间（决定 R4）。
- c. 智能体编排（群组节点）的实际可用性，需控制台验证（标 🔧 项）。
