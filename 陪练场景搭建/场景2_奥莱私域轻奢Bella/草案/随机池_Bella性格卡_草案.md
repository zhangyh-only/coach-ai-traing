# 随机池 · Bella 性格卡（场景2）— 草案

> **用途**：场景2（奥莱私域轻奢尝鲜族 Bella）的角色性格池（personality-pool）4 张卡内容草案。
> **对标**：场景1 `公共_跨场景复用/随机机制/最新随机池内容.md` 的 4 张 Elena 卡（A-理性型 / B-比价型 / C-冷淡型 / D-心动型）。
> **字段契约**（与代码 `NstrRoleplayCard` 对齐）：`{name, content, enabled}`。`name` 仅内部识别、不注入；`content` 在当前 prompt 运行时注入 `{rolePlayPersonality}`；`enabled=false` 即下架。
> **设计原则（复用场景1 随机池约定并吸收 2026-07-01 调整）**：性格卡只写"这一局怎么反应"的行为卡，**不重复画像**（Bella 是谁交给提示词《你是谁》）、不写具体台词、不写固定流程脚本。每张卡用同一组字段表达：说话气质 / 最先关切 / 证据阈值 / 追问策略 / 被打动信号 / 被推开信号。每张卡都必须带"追问后的出口动作"，防止性格特征被模型放大成连续盘问。
> **产品池**：场景2 复用场景1 现有 product-pool（15 款 Tabby/Brooklyn），**不另做**。
> **状态**：🔵 草案，**未灌入任何 config_key**——如何并入全局随机池属改动·高架构决策（见文末），待用户拍板 + ADR 后再正式上线。

## 设计说明（每张卡 = 场景1 性子骨架 × Bella 基底）

| 卡 | 对标场景1 | 性子基调 | 最先在意（换成 Bella 的关切） | 出口动作 |
|---|---|---|---|---|
| **A-精算型** | A-理性型 | 平稳冷静、话少有条理、藏情绪 | 先算**价格边界 / 值不值**，正品只顺带确认 | 价格够判断后，转试背/打理/再想想 |
| **B-比价型** | B-比价型 | 较真、话稍多、不轻易服气 | 要一个**为什么选奥莱这只**的综合理由 | 理由接住后，转颜色/上身/决定 |
| **C-怕踩坑型** | C-冷淡型 | 拘谨慢热、话少、被推就躲 | 每局只先冒一个**安全感顾虑** | 追过后，转试一下/再看看/再想想 |
| **D-心动型** | D-心动型 | 热络话多、藏不住喜欢、推进快 | 先看**好不好看 / 上身能不能接住日常** | 被接住后，转活动/打理/收尾 |

> v0.3 的重点不是增加更多角色，而是降低每张卡的极端触发词：A 不变砍价，B 不变抬杠，C 不变全风险盘问，D 不变补全清单。每张卡都要有“问完后往哪走”的出口动作。

## ⚠️ 如何并入全局随机池（改动·高，待拍板 + ADR）

当前随机池 SSOT 标题为「场景一 …」、config_key 全局单一（`nstr.roleplay.personality-pool`），**无按场景路由机制**。Bella 4 卡不能直接塞进同一池（会与 Elena 卡混抽）。落地需在以下方案里选一个，再走 ADR + 改 SSOT/guard + 后端配合：

- **方案A（推荐）独立 config_key**：新建 `nstr.roleplay.personality-pool.scene2`（或约定命名）装 Bella 4 卡；产品池复用场景1 同一 key。后端按场景 ID 选性格池 key 注入。隔离干净，改动集中在"加一个池 + 后端按场景选 key"。
- **方案B 加场景字段**：`NstrRoleplayCard` 加 `scene` 字段，一池混装、后端按 scene 过滤。改后端数据结构，较重、易错。
- **方案C 先草案后灌**：本卡先就绪、暂不进 SSOT，待后端多场景路由方案定了再灌（当前状态）。
