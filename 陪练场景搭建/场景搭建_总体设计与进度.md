## 2. 三链架构总览

被替代的现状：**单体全量系统提示词 + input 直调大模型** → 角色扮演/阶段推进/评估打分互相挤压，扛不住四诉求。
解法：**百炼多智能体 + 知识库 RAG，按职责拆三链**。

| 链路 | 承接的客户诉求 | 主要引用 KB | 调用方式 | 输出 |
|------|----------------|-------------|----------|------|
| **① 角色扮演对话** | ①角色丰富度 ②角色差异性 ④对话推进 | KB4(画像/异议/潜台词) + KB2(产品FAQ) + KB3(阶段动作) + KB1(品牌/红线) | 按需 | Markdown 对话 |
| **② 实时评估** | ③评估准确（即时反馈，不出总分） | KB5(维度判断/红线) + KB3(当前阶段) + KB2/KB3(事实口径) | 必定 | Markdown 即时提示 |
| **③ 整体评估** | ③评估准确（结束出完整报告） | KB5(四件套) + KB3/KB4(动作与画像) | 必定 | Markdown 评分报告 |

> 本期**只搭这三链**，不做对话采集、不做变量生成（与太保项目的范围差异）。

## 3. 全局设计约定（跨场景通用，逐步沉淀）

> 这些约定一旦敲定，所有场景的三链都遵守；是"批量复制场景"前必须先定死的地基。

### 3.0 场景定义（最根本约定）
**一个场景的身份 = 主题 × 画像（具体角色）× 渠道 + 阶段**：
- **主题**（练什么）：价格异议 / 产品知识 / 推荐 / 售后 / 竞品 / 送礼…
- **画像**（对谁练）：质感自用 / 性价比入门 / 礼品选购 / 都市悦己，落成**固定具体角色**
- **渠道**：正价(Retail) / 奥莱(Outlet)（私域二期）。**独立维度，但给"典型搭配"默认**——质感自用·都市悦己 → 默认正价；性价比入门 → 默认奥莱；**允许例外**（如正价店遇爱砍价顾客，本身是另一种练法）
- **阶段**（哪个环节）：Coach 销售流程＝迎宾破冰→需求诊断→触感体验→价值传递→主动成交（→售后维系，本期不含）。**本次 Coach 场景默认走「全流程」**（迎宾破冰→主动成交，到成交为止）；**主题是全流程里的"重点考察环节"**（场景1 全流程，重点在价值传递/成交段的价格异议）。
  - **阶段的双视角（重要）**：阶段本质是**销售流程视角**（销售 SOP ＋ 评估"流程意识"的标尺）。① 进**角色扮演链**时翻译成**顾客视角**——顾客不知道、绝不说出"价值传递"等阶段名，只表现对应的**心理状态＋反应信号**，阶段推进由顾客**外显信号**自然带出（非顾客宣布）。② 进**评估链**时才是**销售视角**的流程标尺（评有没有按阶段走/跳步）。
**不算场景身份**（不进命名、不决定"是哪个场景"）：产品（软背景）、过招剧本骨架（破雷同机制）、互动模式 / 难度（修饰）。
**命名规则**：`场景N_主题_画像角色`（例：场景1_价格异议_质感自用Elena）。**复制新场景 = 换"主题 / 画像 / 渠道 / 阶段"任一要素。**

| 约定项 | 当前结论 | 状态 |
|--------|----------|------|
| **角色实例化方式** | **定稿（对抗评审后）**：身份固定 Elena；破"每次差不多"主力＝开局随机抽 1 条**「过招剧本骨架」**（异议顺序/松口/拉扯/成交路径不同，工作流 random + 最近 N 局不重复）+ 异议 RAG + 响应式；背景/情绪降调味。**适度、不过度随机**（4 骨架共享 Elena/价格异议统一底色）。详见 [专题 v2](专题_角色多样性与随机机制设计.md) + [样例](场景1_价格异议_质感自用Elena/过招剧本设计样例.md) | ✅ |
| **防护模块繁简度** | **已定**：身份防护+行为边界保留**精简版**（核心铁律去重，约原 1/3 篇幅）；**单一焦点软化**为风格倾向（鼓励一次聚焦一个关切、允许自然追问/带情绪/带铺垫），**废除"机械数问号自检"**；新增正向人设指令"你是有血有肉、会犹豫会被打动的真实顾客"，过度"禁止"转"应该怎样"，给角色活度让空间 | ✅ |
| **阶段推进条件写法** | **已定**：心理70%+行为30%写成**可观测信号**——心理用顾客外显语言/情绪（质疑转松动、主动问使用场景/搭配、追问价格细节），行为用销售具体动作（给出基于材质/工艺的价值理由），杜绝"销售介绍完"式模糊；每阶段 progression_limits 最大轮次强制推进；收尾导购送客触发制 | ✅ |
| **异议库格式** | 按阶段挂，每阶段 **3–5 个**，格式「**关切主题：关切内容**」（不写完整台词）；开场阶段不出深层价格质疑 | ✅ |
| **收尾机制** | **导购送客触发制**（防 AI 顾客抢先结束）；`stage_max` 触发收尾 | ✅ |
| **自定义变量命名约定** | 待定（渠道/阶段/画像/产品/主题/互动模式/难度 的变量名与取值枚举） | ⬜ |
| **KB 检索规则写法** | 在每链系统提示词写明"何时查、查哪个标签范围"；按 §2 调用方式（必定/按需） | 🔶 |
| **输出格式规范** | 全 Markdown；整体评估用标题+表格排版；展示 KB 来源角标 | ✅ |
| **事实层/表达层分离** | 事实(产品/价格/材质/政策)走 RAG 必定召回/插件取数**锁死防幻觉**；表达(顾客话术/评估建议)**放开**口语情绪不钉字数。见[百炼报告 §4.1](百炼空间能力机制分析报告.md) | ✅ |
| **内容 RAG 化** | 异议库/画像细节/产品 FAQ **沉知识库按需检索**（标签:画像/阶段/产品），不全量内联进 prompt | ✅ |
| **产品绑定方式** | **软背景，非写死单品**（"在看包、对某类有意向"），导购推荐他款顾客自然响应多品类——正面解 R2「困在单品」 | ✅ |

## 4. 链路一 · 角色扮演对话（AI 顾客）设计

> 📄 **场景1 草案**：[链一_角色扮演_系统提示词草案.md](场景1_价格异议_质感自用Elena/链一_角色扮演_系统提示词草案.md)（**v3 机制化回炉版** · 🔵 待 review）：Agent 2.0 + 异议/产品/画像 **RAG 按需检索** + **产品软绑定** + 百炼机制配置区

**承接**：①角色丰富度 ②角色差异性 ④对话推进。
**百炼配置**：Agent 2.0；知识库按需（KB4/KB2/KB3/KB1）；自定义变量注入场景参数；短期记忆 ≥20 轮。

### 4.1 系统提示词结构骨架（8 模块） — 🔶 已呈现待最终确认

| # | 模块 | 职责 | 来源 |
|---|------|------|------|
| 0 | **核心行为协议** | 顾客身份铁律、防角色反转、单一焦点（每轮一话题一问）、不暴露 AI | 历史场景复用·**需精简** |
| 1 | **角色设定 characterInfos** | 姓名/性别/年龄/职业 + 详细 `character_profile`（基本信息·消费经历·性格·痛点·语言特征·行为模式） | 变量 + KB4-质感自用「角色扮演行为指引」 |
| 2 | **情境 systemBackgroud** | 时空/客户状态/历史背景/触发因素 + Coach 固定约束（有实物可试背、反感贴身跟随、无电话式戒备） | 变量 + 固定内置 |
| 3 | **阶段状态机** | 阶段范围（场景1=价值传递→主动成交）；每阶段五要素（停留指导/内心状态/行为特征/核心关注/推进条件）；推进条件 心理70%+行为30%；强制推进 + 送客触发收尾 | 初始输入 §3 + KB3-流程 |
| 4 | **异议库 objection_library** | 按阶段挂 3–5 个，格式「关切主题：关切内容」；开场不出深层价格质疑 | KB4-异议阶段分布 + 潜台词 |
| 5 | **产品 FAQ 与回应边界** | 被问产品才按需检索；顾客只表达感受/疑虑/追问，不主动介绍产品 | 按需 KB2-Tabby/皮革 |
| 6 | **KB 检索规则** | 写明何时查、查哪标签范围（必定/按需） | 初始输入 §2 |
| 7 | **自定义变量占位** | 渠道/阶段/画像/产品/主题/互动模式/难度 —— 只放"选择项" | 初始输入 §6 |

### 4.2 待逐模块敲定的争议点

1. ✅ **已定** 模块1 角色实例化：单场景写死具体角色（Elena），profile 全字段写死；差异性靠批量阶段生成多个实例。字段结构沿用：基本信息·消费经历·性格·痛点需求·语言特征·行为模式。
2. ✅ **已定** 模块0 防护繁简度：精简身份防护+行为边界，单一焦点软化为风格倾向（废除机械数问号），新增正向人设指令平衡防御性。
3. ✅ **已定** 模块3 推进条件：写成顾客侧可观测外显信号(70%)+销售具体动作(30%)，progression_limits 强制推进，送客触发收尾。

### 4.3 本链需扛住的 POC 硬性要求

- **R1 角色要活**：会犹豫/追问/有情绪/被话术推开或拉近；禁止对报价只回"好的"、禁止死板走完流程。
- **R2 不困在单品**：可接受合理推荐其他款，不把推荐都拉回 Tabby；覆盖多品类。
- **R6 活人感**：语气贴 GenZ、有朋友感，不官方不机器化。
- 开场不冷漠：体现正常人际礼貌与配合度。

## 5. 链路二 · 实时评估 设计

**承接**：③评估准确（即时反馈）。
**定位**：对练过程中按轮/分段触发，给即时信号（本轮异议处理到位？阶段推进了？踩红线了？），**不出总分**。
**百炼配置**：Agent 2.0，独立 agent；业务系统按轮/分段调用；知识库**必定**（KB5 维度判断 + 红线）；短期记忆看最近几轮；输出 Markdown 即时提示文本。
**内容规格**：输出「即时观察 + 改进提示」，引用 KB5 维度"好/差判断逻辑"与红线（夸大/踩 GenZ 禁忌/贬低 Outlet）。
**POC 要求**：R3 话术打动力（指出表达问题、给可直接用方向）；R4 价格口径准（须知折扣/活动价，无主数据则标待补、不臆造）；R7 产品知识低分可指向对应 KB 资料。

> 详细规格 ⬜ 待角色扮演链定稿后对齐。

## 6. 链路三 · 整体评估 设计

**承接**：③评估准确（完整报告）。
**定位**：对练结束，按 KB5 四维度出完整评分报告 + 改进建议。
**百炼配置**：Agent 2.0，独立 agent；业务系统结束时调用；知识库**必定**（KB5 四件套）；读全程对话历史；输出 Markdown 评分报告（维度/得分/档位/证据/建议，标题+表格排版）。

### 6.1 四维度（Coach POC 定稿）— ✅

| 维度 | 副标题（**勿据字面误解**） | 权重 |
|------|---------------------------|------|
| 专业性 | 知识运用与价值传递 | **30** |
| 同理心 | **需求挖掘与场景锚定**（≠单纯情绪共情） | **25** |
| 沟通技巧 | 共情与引导 | **25** |
| 流程意识 | 节奏与流程把控 | **20** |

- 每维度 **2–4 个指标**：指标名 / 分值 / **评估标准三层**（概念层 + 评估要点用疑问句 + 应用层结合 Coach 场景）。
- **总分严格 = 100**；维度三档（优秀/良好/待提升，区间见 KB5-评分）。
- **禁止指标**（硬约束）：时间限制 / 语音技术（语速语调停顿）/ 机械话术（必须说 X）/ 表情动作 / 假设结果。
- **证据导向**：每个扣分指向对话具体片段；红线落对应维度扣分（事实类→专业性；沟通禁忌→沟通技巧；流程跳步→流程意识）。

> 指标细化与评估标准三层 ⬜ 待对齐。

## 8. 历史风险规避台账（太保踩过的坑，逐条主动规避）

| 坑 | 规避动作 | 状态 |
|----|----------|------|
| 异议与阶段不匹配 | 异议按阶段挂（KB4-阶段分布），开场不出深层价格质疑 | ✅ 已入约定 |
| 异议数量过多 | 每阶段 3–5 个 | ✅ |

## 12. 标准产品借鉴台账（"全量prompt+直调"标准版经验的吸收与规避）

> 来源：`/Users/zhangyh/Documents/基础提示词`（标准产品三链：系统模板 + 太保实例）。标准产品同为"全量提示词+直调大模型"，有业务抓手但存在**角色不真实 / 话术僵硬 / 不像真人**问题。
> 角色扮演链精读 672 行系统模板；评估两链分析由 workflow `wm0rknsir` 补充（见 §12.5 实时评估 / §12.6 整体评估）。

### 12.1 核心洞察（架构级，最重要）
标准版把整套「**反应控制引擎**」（4 类销售信号识别 → 质量评级 → 4 维反应调节 → 多维整合 → 8 步输出检查）**塞进了角色扮演链**——顾客每说一句话前都要先给销售打一遍分，必然像填表、不像真人。这是"话术僵硬"的**结构性根因**。
→ **Coach 三链架构的结构性优势**：把"销售行为质量判定"归**评估链**（KB5），角色扮演链只管"演好这个顾客"（被打动就软化、被推开就冷淡），天然减负 → 更自然。**印证初始输入"单体 prompt 职责互相挤压"的诊断。**

### 12.2 值得借鉴的业务抓手 → 用到 Elena
| 抓手 | 用法 |
|------|------|
| 禁止暴露内部机制（不输出阶段判定/异议选择/推进逻辑） | ➕ 加进模块0 |
| 收尾两段式（第1轮释放**明显**结束信号给导购"收尾窗口"，第2轮才告别） | ➕ 细化模块3收尾（落地"导购送客触发制"） |
| 异议智能选择（针对导购当前话语、**排除已答的**、**禁止照搬库原文**、改写成自然反应） | ➕ 加进模块4使用规则 |
| 追问硬上限（同一问题点有上限，防无限刁难） | ➕ Elena 懂行版放宽到"答空洞才追第2次" |
| 话题切换衔接（先回应上文+过渡词+新话题，不硬抛新问题） | ➕ 加进模块0风格 |
| 变量化字段打包（phases = 阶段名+阶段动作+maxRounds+异议，整体打包） | ✅ 对齐模块7变量清单 |

### 12.3 必须规避的僵硬根因（反面教材）
| 反面写法 | 为何对 Coach 有害 |
|----------|-------------------|
| **"你是普通顾客不是专家，禁用专业术语、禁止指出销售错误、不懂就问"** | ⚠️**最关键**：对太保电话保险小白合理，但 **Elena 是懂行、会盘问材质工艺的高客单顾客**——套这条 Elena 就废了。Coach 高客单场景的**本质差异**，反证"懂行追问"设计是对的 |
| 8 步评级 + 4 维反应引擎 | 角色扮演链不该背，搬给评估链（架构优势） |
| "执行检查：数问号、超1个必删" | 机械计数 → 僵硬（已废除） |
| "绝对禁止附和：不准说'好的/听起来不错/有道理'" | 让顾客冷漠戒备、不像真人；Elena 有礼貌，正常认可该保留 |
| 写死"配合型客户"单一类型 | Coach 靠**画像**驱动差异，不用单一客户类型 |

### 12.4 变量字段对照（标准产品 FreeMarker → Coach 自定义变量）
- 标准产品：`bot.display.name`(角色名) / `bot.ext.description`(描述) / `extra.company_name`(公司) / `extra.product_name`(产品) / `bot.display.prompts`(画像 Role) / `base.background`(场景背景) / `context.phases`(阶段:phaseName+botAction+maxRounds+phaseProblems异议)。
- → Coach 模块7：画像(Elena 写死) / 产品(Tabby26) / 渠道(Retail) / 阶段范围 / 主题 / 互动模式 / 难度。**借鉴**：标准版把"阶段动作+轮次上限+异议"随 phases 一起打包的做法，值得用在阶段状态机的参数化上。

### 12.5 实时评估链借鉴
**定位（借鉴）**：只评**最新一条销售消息**，不评历史、不总结多轮、不跨节点；三步法 定位评估对象 → 匹配维度 → 综合评估。输出"即时观察 + 改进提示（+参考话术方向）"，简短。
**规避根因**：
- 强字数上限（≤100字）压垮语气 → 电报腔：实时提示要短，但**别钉死字数**，保留口语呼吸感。
- "参考话术必须照资料标准表述 + 不得编造" → 说明书腔：**KB 里要存口语化标准话术**，而非纯书面参数（对 KB3/KB5 的改进提示）。
- 自查标签 / 三段式公式化 → 模板味：自查放内部，别污染输出。

### 12.6 整体评估链借鉴
**借鉴抓手**：
- **三级评分卡**：维度 → 指标 → 评估要点(疑问句) + **显式分值组成**（如报价维度=车险准确10+非车完整10+衔接5），可量化、可追溯。← 对齐 KB5 维度结构。
- **完成度 → 分数段映射**（全完成→满分 / 80%+→90-100% / 一半→50-80% / 少量→0-40%）。
- **培训场景宽容评分**（陪练非考核、鼓励而非打击、核心要点完成即高分）。
- ⭐ **RAG 事实溯源红线**：检索内容 = 产品事实**唯一来源**；宁可建议笼统不可细节编造；评估标准里的事实示例 ≠ 事实依据，与检索冲突一律以检索为准。
- ⭐ **评分维度 vs 事实维度 双轨**：判"该不该说"用评分卡、判"说得对不对"用 RAG，两者解耦、不互相加码。
- **红线去重**：销售已说过的（即使换词）禁止再作为建议。
- **结构化输出**：分维度 + 证据 + 建议（Coach 用 Markdown 标题/表格，不用 JSON）。
**规避根因**：
- ⭐ **画像写成静态长段、无情绪状态机** → 客户平铺复述设定、不像真人：**画像要带随对话演化的情绪/立场**（Elena 草案模块3阶段情绪曲线已体现，继续强化）。
- ⭐ **表达层被事实锁死却没留口语/共情空间** → 同义改写僵硬：**事实层(RAG锁死) vs 表达层(口语/情绪自由) 显式分离**——解决"评估建议/话术僵硬"的总钥匙。
- 字数句式钉死、等价词典静态枚举（查表式）→ 规避。

> **跨链总结论（升为全局设计原则）**：解决"建议/话术僵硬" = 事实层用 RAG 锁死防幻觉 + 表达层放开口语情绪；**KB 应存"口语化标准话术"而非纯书面参数**。
