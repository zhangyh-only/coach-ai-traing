# 角色扮演提示词组成 & 性格/产品随机机制 简要说明

> 范围：仅「机器人回复」链路（AI 顾客的回答）。整体点评、单轮点评不涉及。

## 一、提示词怎么拼出来——两种方式，自动切换

机器人回复走阿里云百炼 agent。给 agent 的提示词有两种组装方式，由「这个陪练有没有声明 `customParams`」自动切换：

| | 方式一（标准） | 方式二（自定义变量） |
|---|---|---|
| 固定骨架放哪 | 本地（程序拼整段） | 阿里云 agent 应用里 |
| 程序传给 agent | 一个变量 = 完整提示词 | 多个自定义变量（性格、产品…） |
| agent 模板 | 只有 `${nstr_action_prompt}` | 固定骨架 + `${roleplay_personality}`、`${roleplay_product}` |
| 触发条件 | `customParams` 为空 | `customParams` 非空 |

**切换开关**：agent 调用配置 `nstr.agent-call.params.bot-answer` 里的 `customParams` 是否非空。
- 不配 / 为空 → 方式一（与老陪练一致，不破坏现状）。
- 配了变量名清单 → 方式二（按声明取自定义变量）。

## 二、性格、产品的随机性设计

- **两件独立的事**：性格、产品各有一个池，各自独立随机抽、互不影响——一个池为空不耽误另一个。
- **看 `customParams` 里有什么就抽什么**：声明了 `roleplay_personality` 才抽性格、声明了 `roleplay_product` 才抽产品；没声明的池根本不读。将来加新变量，照同一套格式扩展即可。
- **整局锁定（幂等）**：抽到后把「实际文本」按 `record_id` 存进 `nstr_prompt_replace_text`（`order_no=0`）；同一局后续每轮、以及「重新生成」都复用同一结果，不会中途换人换包。
- **兜底不阻塞**：池为空 / 抽不到 / 读库异常 → 用默认中性文案，开局照常。

## 三、对应的池（都在 `nstr_global_config`）

| 变量名（user_prompt_params 的 key） | 池 config_key | 内容 |
|---|---|---|
| `roleplay_personality` | `nstr.roleplay.personality-pool` | 性格池：本局过招风格（如 A理性盘问 / B横向对比 / C防御冷淡 / D心动纠结） |
| `roleplay_product` | `nstr.roleplay.product-pool` | 产品池：一进店看上的那只包的外观/款式描述 |

**池内每个对象只有三个字段**：

```json
{ "name": "识别名", "content": "实际注入提示词的文本", "enabled": true }
```

- `name`：运营识别用，**不喂 AI、抽卡不用**。
- `content`：真正注入提示词的文本。性格 = 整段过招风格行为指令；产品 = 外观/款式描述（不报型号、不含价格）。
- `enabled`：`false` 不参与抽取。

> 锁定到 `nstr_prompt_replace_text.val` 的是 `content` 全文（不是 id / 下标），所以池里增删卡片不会让历史局错位。

## 四、一局完整流程（以陪练 492 为例）

1. 学员开口 → 机器人回复链路。
2. 读 `nstr.bot-display.call-service` = `aliyun-agent` → 走百炼 agent。
3. 读 `nstr.agent-call.params.bot-answer` 的 `customParams = ["roleplay_personality","roleplay_product"]` → 非空 → 方式二。
4. 对每个声明的变量：先查本局是否已锁定 → 没锁就从对应池随机抽一个 `enabled` 卡 → 落库锁定。
5. 把 `{"roleplay_personality":"<本局性格文本>","roleplay_product":"<本局产品文本>"}` 作为 `user_prompt_params` 传给 agent。
6. agent 用它内置的固定骨架 + 这两个变量，生成 AI 顾客这一句回答。

## 五、关键代码落点

| 职责 | 位置 |
|---|---|
| 变量名 / 池 key / 默认文案 常量 | `NstrRoleplayConstants` |
| 卡片格式（性格、产品共用） | `NstrRoleplayCard`（`name`/`content`/`enabled`） |
| 抽卡 + 锁定 + 按 `customParams` 取值 | `NstrRoleplayCustomVariableHandler#resolveRoleParams` |
| 方式一/方式二判断 + 拼 user_prompt_params | `NstrBotDisplayCallServiceImpl#getBotConfig`（ALIYUN_AGENT 分支） |

> 数据侧需配套：方式二的固定骨架要贴进阿里云 agent 应用、留 `${roleplay_personality}`/`${roleplay_product}` 两个口子；性格池/产品池/customParams 配在 `nstr_global_config`（见 `coach-base-1.sql`）。
