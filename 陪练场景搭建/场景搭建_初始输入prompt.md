# Coach 陪练场景搭建 · 初始输入 Prompt（v1.0）

> 用途：作为"开启 Coach 陪练场景搭建"阶段的初始输入，喂给负责搭建的 AI（或本人作业），框定**目标、平台底座、三链架构、内容规格、POC 要求、首批场景**。
> 上游已完成：Coach 陪练知识库 45 篇（`知识库搭建/具体KB内容/`，KB1–KB5）。
> 本阶段**只搭三条链路：①角色扮演对话 ②实时评估 ③整体评估**；**不做对话采集、不做变量生成**（这是与太保项目的范围差异）。

---

## 0. 目标、范围与为什么换架构

**目标**：在**阿里云百炼**平台搭建 Coach 门店导购陪练，达成客户四项诉求——
**①角色丰富度（对话中可体现）②角色差异性 ③评估准确 ④对话推进顺畅**。

**为什么必须换架构**：现状线上系统是"**一个全量系统提示词（所有逻辑塞在一起）+ input 直调大模型**"。这种单体 prompt 让角色扮演、阶段推进、评估打分的职责互相挤压干扰，**结构上就扛不住上面四项诉求**——角色千人一面、阶段推进失控、评估不稳。
**解法 = 百炼多智能体 + 知识库 RAG，按职责拆成三条链**，各司其职、各自引用对应 KB。这与太保项目"采集→变量生成→陪练→评估"分节点的思路同源，只是本次 Coach **直接搭运行期三链**，不做前置的采集与变量生成。

**本期范围（只搭这三条）**：
1. **角色扮演对话**链 → 承接 ①角色丰富度 ②角色差异性 ④对话推进
2. **实时评估**链 → 承接 ③评估准确（对练过程中即时反馈）
3. **整体评估**链 → 承接 ③评估准确（对练结束出完整报告）

## 1. 平台能力底座（阿里云百炼，已据官方文档核实）

> 来源：百炼《新版智能体应用》官方文档 help.aliyun.com/zh/model-studio/new-single-agent-application

| 能力 | 要点 | 对三链的用法 |
|------|------|------------|
| **智能体应用 Agent 2.0** | 知识库+MCP 统一为"工具"，由智能体**自主规划调用**；展示"规划-执行-反思"全过程 | **三链统一用 Agent 2.0（已定）** |
| **知识库 RAG** | 文档/表格/图片/音视频；高精检索；**调用方式可选"必定/按需"**；支持**标签限定检索范围**；系统提示词中定义检索规则 | KB 上传后被各链按需/必定检索（见第 2 节对接表） |
| **自定义变量** | 系统提示词"新建变量"，提示词内 `/` 引用 | **场景参数化的核心手段**——用变量把"渠道/画像/产品/阶段/主题/互动模式"注入角色扮演链（替代"变量生成"） |
| **记忆** | 短期记忆 0–30 轮（0=不传递）；**长期记忆暂不支持** | 角色扮演链短期记忆建议 ≥20 轮保人设连贯；评估链需读对话历史 |
| **发布与编排** | 三链**各自发布为独立 agent（共 3 个）** | **触发与编排由业务系统程序控制**——业务系统按需调用百炼 agent，**不在百炼内部编排**（已定） |
| **回复增强/干预** | 异常用固定话术干预；模型切换增强；展示来源 | 角色扮演异常兜底；评估展示 KB 来源角标 |
| **输出形态** | **无结构化输出（JSON），三链输出全部为 Markdown 文本（已定）** | 评估报告用 Markdown 结构化排版（标题/表格/分点），不依赖 JSON 解析 |

> **平台决策已锁定（开工版）**：① 三链统一 **Agent 2.0**；② 知识库**单库+标签**；③ 输出**全部 Markdown 文本**；④ 三链发布为 **3 个独立 agent**，**触发与编排由业务系统控制**（百炼侧只管单 agent 的能力，不做跨 agent 编排）。

## 2. 知识库上线与三链对接（45 篇 KB → 百炼知识库）

## 3. 三条链路设计（含各自"内容规格"）

> 内容规格借用同平台已验证的变量字段（太保"变量生成规则 json"，**平台级、Coach 同平台可直接复用**），但本次它们不是"变量生成"的产物，而是**直接写进三链系统提示词/配置的内容规格**。

### 链路一 · 角色扮演对话（AI 顾客）
**承接**：①角色丰富度 ②角色差异性 ④对话推进。
**百炼配置**：Agent 2.0；知识库按需（KB4/KB2/KB3/KB1）；**自定义变量**注入场景参数；短期记忆 ≥20 轮（保人设/阶段连贯）。
**系统提示词要承载的内容规格**：
- **角色设定（对应 characterInfos）**：姓名/性别/基本背景 + **详细画像 character_profile**（基本信息·消费经历·性格特点·痛点需求·语言特征·行为模式）← 取自 KB4 对应画像的"角色扮演行为指引"。
  - **角色丰富度**：character_profile 必须结构化、立体（不是一句话标签）。
  - **角色差异性**：同一画像类型下，每个场景生成**有差异的具体角色**（年龄/职业/品类偏好/语气不同），破"千人一面"。
- **情境（systemBackgroud）**：时空环境/客户状态/历史背景/触发因素。**Coach 约束**（区别于太保电话场景）：线下门店**有实物、可试背触摸**；私域**短链路、低打扰**；开场顾客对导购无电话式强戒备，但反感贴身跟随。
- **阶段状态机（stage_definitions + stage_progression_limits + stage_max）**——**这是"对话推进"的核心机制**：
  - 每阶段五要素：停留指导/内心状态/行为特征/核心关注/**推进条件**。
  - **推进条件 = 客户心理状态变化为主(约70%) + 销售关键行为为辅(约30%)**（太保踩坑结论：避免"销售完成介绍后"这类模糊条件）。
  - `stage_progression_limits` 设每阶段最大轮次**强制推进、防止卡阶段**；`stage_max` 触发收尾，收尾用"**导购送客触发制**"（防 AI 顾客抢先结束）。
  - 阶段对齐 KB3 线下正价店 6 阶段（迎宾破冰→需求诊断→触感体验→价值传递→主动成交→售后维系），单阶段精练或全流程。
- **异议库（objection_library）**：按阶段组织，**每阶段 3–5 个**，格式"**关切主题：关切内容**"（不写完整台词）← KB4 异议总表/阶段分布。开场阶段不出深层价格质疑（还没报价）。
- **产品 FAQ 与回应边界** ← KB2（被问到才按需 RAG）。
**POC 硬性要求落到本链**：
- **R1 角色要"活"**：会犹豫、会追问、有情绪、会被话术推开/拉近；**禁止对报价只回"好的"、禁止死板走完设定流程**。
- **R2 不困在单品**：即便设定有心仪款，也能接受导购合理推荐其他款；**不要把所有推荐都拉回 Tabby**；画像/产品覆盖多品类。
- **R6 活人感/生活化**：语气贴 GenZ、有朋友感，不官方、不机器化（KB1-沟通 + KB3-GenZ活人感沟通法则）。
- 开场不冷漠：体现正常人际礼貌与配合度。

### 链路二 · 实时评估（对练过程中即时评）
**承接**：③评估准确（即时反馈）；缓解 POC"话术不够打动""不知怎么和 AI 互动"。
**定位**：对练过程中**按轮/按分段触发**，给**即时信号**（本轮异议处理到位吗？阶段推进了吗？踩红线了吗？），**不出总分**。
**百炼配置**：**Agent 2.0**，发布为**独立 agent**；**由业务系统在对练过程中按轮/分段调用**（触发由业务程序控制）；知识库**必定**（KB5 维度判断逻辑 + 红线）；短期记忆（看最近几轮对话）；**输出为 Markdown 即时提示文本**（无 JSON）。
**内容规格**：轻量评估——输出"**即时观察 + 改进提示**"，例如"价格异议未给专业理由，建议从来源/工艺/检验切入"。引用 KB5 维度的"好/差判断逻辑"与红线（夸大/踩 GenZ 禁忌/贬低 Outlet）。
**POC 要求**：**R3 话术打动力**（不只判正确，要指出表达问题、给可直接用的方向）；**R4 价格口径准**（须知道折扣/活动价，无主数据则标待补、不臆造）；**R7 产品知识低分可指向对应 KB 资料**。

### 链路三 · 整体评估（对练结束出报告）
**承接**：③评估准确（完整报告）。
**定位**：对练结束，**按 KB5 四维度出完整评分报告 + 改进建议**。
**百炼配置**：**Agent 2.0**，发布为**独立 agent**；**由业务系统在对练结束时调用**；知识库**必定**（KB5 四件套）；读**全程对话历史**；**输出为 Markdown 评分报告**（维度/得分/档位/证据/建议，用标题+表格排版，无 JSON）。
**内容规格（对应 evaluationInfos）**：
- **四维度（Coach POC 定稿）**：专业性—知识运用与价值传递(**30**) / 同理心—**需求挖掘与场景锚定**(**25**) / 沟通技巧—共情与引导(**25**) / 流程意识—节奏与流程把控(**20**)。
  > 注意官方副标题，**"同理心"= 需求挖掘与场景锚定，不是单纯情绪共情**；勿据字面误解。
- 每维度 **2–4 个指标**：指标名 / 分值 / **评估标准三层**（概念层 + 评估要点用疑问句 + 应用层结合 Coach 场景判断）。
- **总分严格 = 100**；维度三档（优秀/良好/待提升，区间见 KB5-评分）。
- **禁止指标**（硬约束）：时间限制 / 语音技术（语速语调停顿）/ 机械话术（必须说 X）/ 表情动作 / 假设结果。
- **证据导向**：每个扣分指向对话具体片段；**红线落对应维度扣分**（事实类→专业性；沟通禁忌→沟通技巧；流程跳步→流程意识）。
**POC 要求**：**R3 评分可靠可解释**（呼应 Q3"评价准确性"40.8%）；评估要**指出学员表达上的问题**，而非只给分。

## 4. 客户四诉求 → 落点对照（验收锚点）

| 客户诉求 | 落到哪条链 / 哪个机制 |
|----------|---------------------|
| **角色丰富度（对话中体现）** | 角色扮演：结构化 character_profile（KB4 角色扮演行为指引）+ 短期记忆保持人设 |
| **角色差异性** | 角色扮演：同画像下生成差异化具体角色 + 自定义变量参数化场景 |
| **评估准确** | 实时评估 + 整体评估：KB5 维度判断逻辑/指标/评估标准三层/总分100/禁止指标/证据导向 |
| **对话推进顺畅** | 角色扮演：stage_definitions 推进条件(心理70%+行为30%) + progression_limits 强制推进 + 收尾送客触发 |

## 5. POC 反馈 → 设计要求与取舍台账（两轮：首轮 52 份 + 第二轮 260 份）

> 数据经逐项核对原始 xlsx；80% 用户评 4–5 分，整体认可度高。涉及百分比注明轮次。

**A. 硬性设计要求（场景设计可控，每个场景都要扛住）**
- **R1 角色要活**（第二轮 Q3 真实感 49% 居首；原话"对报价只回'好的'""缺真实犹豫追问""完全按设置走缺乏真实性"；首轮"AI 模式很固定总引导到 Tabby 展/装 iPad，不随提问深入"）→ 见链一。
- **R2 不困在单品/多品类**（原话"AI 被设定买 Tabby，推荐其他款被拉回""推荐针对性只有 Tabby"；Q5"不限定购物框架"12 次）→ 见链一。
- **R3 评估打动力 + 可靠性**（Q3 评价实用性 43% + 评价准确性 40.8%；Q4"评价正确但话术不够打动顾客"37%；"评价没指出我表达上的问题"）→ 见链二/三。
- **R4 价格/活动口径准**（"AI 对已折扣商品未知，价格评价不准确"）→ 见链二。
- **R5 场景入口清晰 + 顾客类型可选/随机**（第二轮"找不到入口"10.8%；**首轮同类高达 31%**；Q5"顾客类型可选或随机"）→ 平台/产品侧 + 自定义变量支持。
- **R6 活人感/生活化**（"太官方""回答太机器化"）→ 见链一。
- **R7 产品知识低分可承接资料**（Q3"得分低直接给正确产品知识培训资料+示范案例"）→ 见链二。

**B. 已知诉求 · 本期取舍台账（第二轮 Q5 开放列 260 行，显式登记，不静默丢弃）**

| 诉求（出现频次） | 本期处理 |
|----------------|----------|
| 上传图片→生成推荐话术/上身效果（16） | 平台能力，**本期记录，待平台支持后接入"带图求购"场景** |

## 8. 搭建顺序与初版验收（借太保初版方法 + 历史风险规避）

**搭建顺序**：一次搭**一个场景的三链配齐**（角色扮演 + 实时评估 + 整体评估）→ 冒烟测试（走通一轮对话+实时+整体评估）→ 功能测试 → 边界测试。
**初版聚焦**（不求全覆盖）：先做 **2–3 个组合**，建议场景 1（正价 Tabby 价格异议·质感自用）+ 场景 2（奥莱比价·性价比入门）+ 场景 3（皮革知识自查）。
**历史风险规避（太保踩过的坑，逐条主动规避）**：
- 异议与阶段不匹配 → 异议按阶段挂（KB4-阶段分布），开场不出深层价格质疑。
- 异议数量过多 → 每阶段 3–5 个。
- 阶段推进条件模糊 → 客户心理 70% + 销售行为 30%。
- 开场冷漠（只回"嗯""哦"）→ 开场体现正常礼貌配合度。
- 评估含不当指标 → 禁时间/语音/机械话术/表情。
- 评估总分 ≠ 100 → 硬约束 = 100，核心维度≥30。
- 收尾过早 → 导购送客触发制。
- 多信息源冲突 → 优先级：上传/显式设定 > 自定义变量 > 知识库。
