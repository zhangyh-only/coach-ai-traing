# COACH 双场景角色扮演链设计实现说明（内部宣导）

日期：2026-07-02

适用范围：场景1「质感自用 Elena」与场景2「奥莱私域轻奢 Bella」的角色扮演链设计、调优、代码配合与验证闭环。

## 一、这份文档讲什么

这份文档用于内部宣导 COACH 两个 AI 顾客场景的设计实现方式。

它重点回答四个问题：

1. 为什么这两个场景没有完全沿用标准 AI 陪练的 Agent / 知识库式搭建方式。
2. 双场景是怎么拆开的，Elena 和 Bella 分别训练什么能力。
3. 中间遇到的重复追问、场景串味、旁白、收尾等问题是怎么定位和解决的。
4. 代码侧如何配合内容侧，把“提示词调优”做成可验证、可回归的系统工程。

一句话结论：

COACH 这次角色扮演链不是单纯“写一份更长的顾客 prompt”，而是采用了“场景画像拆分 + 结构化行为卡 + 后端组装全量 prompt + 直调模型 + 状态账本 + prompt trace + 真实接口回归”的闭环。它牺牲了一部分平台 Agent 托管的便利，换来了更可控的真人感、上下文连续性和问题定位能力。

## 二、和标准 AI 陪练的关键差异

标准 AI 陪练更适合走平台化、配置化路线：Agent 承载链路，知识库提供事实依据，系统通过配置把角色、流程和评估组织起来。COACH 这次的角色扮演链，目标不是让 AI 顾客懂更多产品事实，而是让它在多轮销售接待里更像一个真实顾客。

因此，本次角色扮演链做了几项不同选择：

| 维度 | 标准 AI 陪练常见做法 | COACH 当前做法 | 原因 |
|---|---|---|---|
| 角色扮演载体 | 百炼 Agent 承载角色链路 | 业务后端组装完整 prompt 后直调模型 | 避免 Agent 历史处理和工具编排带来的黑盒影响，保证每轮上下文完整 |
| 知识库使用 | 角色扮演链可查知识库 | AI 顾客不查知识库，事实核查交给评估链 | 顾客查产品库会有上帝视角，容易复述内部卖点，真人感变差 |
| 场景拆分 | 一个通用顾客模板叠加变量 | Elena 正价、Bella 奥莱独立画像和顾虑体系 | 正价顾客和奥莱顾客的心理账户不同，不能混用同一套异议 |
| 随机性格 | 简短性格标签或描述 | 结构化行为卡：说话气质、最先关切、证据阈值、追问策略、被打动信号、被推开信号 | 避免四种性格跑成同一种“问题清单式顾客” |
| 历史记忆 | 主要依赖模型读全历史 | 显式 `<?STATE ...>` 状态账本 + 后端聚合 `rolePlayState` | 重复追问和结尾倒退靠提示词很难完全压住，需要系统性状态源 |
| 排障方式 | 看最终对话结果 | 保存最终 prompt trace，能看到变量替换后的真实入模内容 | 便于区分是提示词问题、变量问题、历史账本问题还是模型随机性 |
| 验证方式 | 人工试聊为主 | 对抗审查、批跑回归、真实接口样本、focused 单测一起做 | 这类问题容易修一处漏一处，需要可复跑的回归闭环 |

这里的核心取舍是：角色扮演链优先服务“演真人”，不让它承担事实核查、产品资料检索、话术打分这些职责。事实正确性和 SA 表现评估由实时评估链、整体评估链承接。

## 三、双场景设计：不是换名字，而是换顾客心理账户

COACH 当前两个角色扮演场景的差异，不是“问不同产品问题”这么简单，而是顾客进入门店时的心理账户不同。

| 场景 | 顾客 | 渠道与语境 | 核心顾虑 | 训练重点 |
|---|---|---|---|---|
| 场景1 | Elena | 正价 Retail 门店 | 这个价位值不值、质感能不能长期用、上身和衣柜是否匹配 | 训练 SA 讲清价值、质感、搭配、长期使用和服务承接 |
| 场景2 | Bella | Outlet 奥莱 / 私域转化语境 | 是不是官方渠道、到手价是否划算、会不会买错或闲置 | 训练 SA 承接正品安心、价格价值、退换和打理顾虑 |

### 1. Elena：正价门店的“值不值”

Elena 的问题应该围绕“我愿不愿意为这只包付这个价位”展开。她自然会问：

- 颜色和衣柜是否搭。
- 版型、皮面、五金是否支撑长期使用。
- 通勤和周末场景是否都能用。
- 价格背后是否有质感、服务和使用频率支撑。

所以 Elena 的模型可见正文里，刻意弱化了“正价”这个词，改用“这个价位 / 值不值 / 为什么选这只 / Coach 门店”。原因是模型一旦反复看到“正价”，容易把它转成顾客台词里的“是不是正价、跟奥莱有什么区别”，导致正价场景被奥莱语义污染。

### 2. Bella：奥莱场景的“放心又划算”

Bella 的问题更像第一次认真买 Coach 的奥莱顾客。她不只是要便宜，而是要确认：

- 是不是官方渠道，是否有小票和售后承接。
- 当前到手价是否足够划算。
- 买回去会不会闲置。
- 材质、清洁、护理、退换是否有兜底。

Bella 的风险在于“奥莱顾虑”很容易变成无限清单：正品问完问小票，小票问完问售后，售后问完问专柜差异，价格问完问还能不能更低。因此 Bella 不是简单写“不要重复问”，而是把顾虑归并为有限的同类关切，并规定什么情况下这类关切算被导购接住。

## 四、角色扮演链的实际运行方式

当前角色扮演链的运行可以理解为五步：

1. 后端读取当前场景的主提示词模板。
2. 后端抽取本局性格卡，注入 `{rolePlayPersonality}`。
3. 后端抽取本局产品卡，注入 `{rolePlayProduct}`。
4. 后端从历史 AI 顾客回复中的 `<?STATE ...>` 聚合本局状态，注入 `{rolePlayState}`。
5. 后端把替换后的完整 prompt 加上全量对话历史，直接调用百炼模型。

模型真正看到的是最终完整文本，而不是一个抽象配置。因此，排查问题时不能只看“模板写得对不对”，还要看变量替换后的最终 prompt 是否对、历史状态是否被正确带入。

### 关键变量

| 变量 | 含义 | 作用 |
|---|---|---|
| `{rolePlayPersonality}` | 本局顾客性格行为卡 | 决定说话气质、最先关切、追问阈值和推进速度 |
| `{rolePlayProduct}` | 本局产品卡 | 锚定顾客正在看的包款、颜色、外观和使用场景 |
| `{rolePlayState}` | 后端聚合的历史状态账本 | 告诉模型哪些关切已经聊过、哪些动作已经做过、当前是否已表态 |

### 内部控制标签

角色扮演链要求 AI 顾客每轮先输出一行状态标签：

```text
<?STATE 关切=...；动作=...；表态=...>
```

这行不是给 SA 看的台词，而是给系统和模型下一轮认历史用的内部账本。它解决的是三个问题：

- 顾客已经问过的同类问题，不要换说法重问。
- 顾客已经做过的动作，不要下一轮又说“我能试背吗”。
- 顾客已经成交、再考虑或不买，不要倒退回继续挑包。

当对话自然结束时，模型需要在告别句后输出：

```text
<?END_CHAT>
```

这个标签帮助业务系统识别角色扮演链已结束。

## 五、从问题到方案：本次调优解决了什么

这次调优不是一次性写完 prompt，而是按真实测试问题逐步收敛。核心问题和处理方式如下：

| 问题 | 现象 | 内容侧处理 | 代码侧配合 | 验证方式 |
|---|---|---|---|---|
| 同类问题反复追问 | 正品、价格、材质、搭配被换说法重复问 | 把顾虑归并为有限关切类，写清“何时算被接住” | `rolePlayState` 聚合同类关切，追加已聊提醒 | Bella 9 局批跑、Elena 6 局主回归 |
| 场景串味 | Elena 被带出奥莱正品、折扣、渠道语义 | Elena 正文去掉高频“正价”字样，改成价值判断表达 | 后端已聊提醒中的“正价差”改为中性“价差” | Elena 6/6 样本顾客回复不含“正价” |
| 状态漏记 | 模型明明问了顾虑，却写 `关切=无` | 加入 `<?STATE>` 与台词一致性规则和反例 | 后端根据正文顾虑词兜底补账 | Elena trace 验证漏记后下一轮已补同类提醒 |
| 动作重复 | 已经试背、看镜子后又重新申请试背 | 动作进入 `<?STATE>` 后视为已发生 | 后端只从 AI 顾客自己的 `<?STATE>` 抽取动作，避免导购话术误写状态 | Bella 状态标签冒烟 2/2 每轮有状态 |
| 旁白心理化 | 括号里写“心里放心、担心、盘算” | 旁白只允许销售看得见的动作、视线、拿包动作 | 回归扫描有效样本是否出现心理括号 | Bella 旁白补跑 3/3 有效样本无心理括号，Elena 6/6 无心理旁白 |
| 产品锚点漂移 | 暖棕、焦糖、太妃棕等颜色被随意改写 | 产品卡颜色只能沿用本局卡片或同一上位色 | 最终 prompt trace 检查产品卡是否替换完整 | 双场景测试未再出现关键颜色漂移 |
| 收尾不稳定 | 顾客说成交或再想想后，导购送客仍不结束 | 成交、再想想、不买都写成两段式收尾 | 结束路径专项回归 | Bella 再想想修补后 2/2 有效样本带 `<?END_CHAT>`；Elena 成交和再想想均跑通 |

## 六、代码配合做了什么

这次代码侧不是“附属开发”，而是角色扮演链稳定性的关键部分。主要配合点包括：

### 1. 最终 prompt trace

涉及类：

- `NstrRoleplayPromptTraceService`
- `NstrBotDisplayCallServiceImpl`
- `NstrToolController`

作用：模型调用前，把变量替换后的最终 prompt 按记录和轮次保存一份短期 trace，排障时可以查询。

它解决的问题是：过去只能看“模板”和“模型输出”，中间的最终入模内容不可见。现在可以确认：

- `{rolePlayPersonality}` 是否真的替换成了本局性格卡。
- `{rolePlayProduct}` 是否真的替换成了本局产品卡。
- `{rolePlayState}` 是否带入了历史关切、动作和表态。
- prompt 里是否残留变量占位或不该出现的工程措辞。

### 2. 自定义变量替换与状态聚合

涉及类：

- `NstrRoleplayCustomVariableHandler`

作用：替换 `rolePlayPersonality`、`rolePlayProduct`、`rolePlayState`，并从历史里聚合状态账本。

这部分的关键不是简单字符串替换，而是把历史对话转成模型下一轮能消费的“已聊事实”：

- Bella 侧聚合正品 / 渠道安心、划算价格、搭配闲置、打理耐用、版型塌变形、容量、退换兜底。
- Elena 侧聚合价格价值、质感耐用、搭配场合、打理售后、版型比例、容量通勤、重量舒适、肩带背法、开合内里、退换。
- 销售输入可以参与判断“某类关切是否被解释过”，但动作和表态只从 AI 顾客自己的 `<?STATE>` 抽取，避免把导购说的话误当成顾客动作。

### 3. 状态漏记兜底

真实回归中发现：即使提示词要求模型每轮写 `<?STATE>`，模型仍可能在正文里问了“链条会不会太重”“颜色会不会太跳”“麂皮打理麻不麻烦”，但状态写成 `关切=无`。

因此后端增加了正文顾虑词兜底：当正文出现“会不会 / 怕 / 担心 / 麻不麻烦 / 值不值”等顾虑信号时，即使 `<?STATE>` 漏记，也会尽量推断到对应关切类，并写入下一轮状态提醒。

这一步很重要：它承认模型不是确定性程序，不能把所有稳定性都压在 prompt 上。内容侧给规则，代码侧补硬护栏，最终靠真实接口验证。

### 4. Focused 单测

本轮不是只靠人工试聊判断“看起来好了”，而是把几个关键风险加了单测：

- prompt trace 保存与查询。
- `rolePlayState` 三字段解析。
- 同类关切聚合。
- 导购开场探询不会过早闭合搭配话题。
- 明确护理回答会闭合打理售后。
- `正价差` 文案不再进入 Elena 最终提醒。
- 模型漏写 `<?STATE>` 关切时，正文兜底能补到账本。

Bella 调优阶段 focused tests 为 5 tests / 0 failures / 0 errors；Elena 调优阶段 focused tests 扩展到 9 tests / 0 failures / 0 errors。

## 七、验证闭环

COACH 双场景最终不是靠“感觉像”验收，而是按四层验证收口。

### 1. 静态审查

检查模型可见正文是否存在不该出现的工程词、变量说明、内部配置话术，以及是否把场景不该有的语义写进正文。例如 Elena 场景重点检查“正品 / 折扣 / 奥莱渠道”是否污染正价顾客表达。

### 2. 对抗审查

针对最容易复发的问题设计 E 组用例：

- 已聊正品后，Bella 是否还换说法问专柜、小票、真假。
- 已聊价格后，Bella 是否继续问还能不能便宜。
- 已试背后，顾客是否还问能不能试背。
- 已成交或再想想后，导购送客是否能触发 `<?END_CHAT>`。
- 最终 prompt 正文是否泄漏“变量、占位、程序、注入、直调”等工程措辞。

### 3. 真实接口批跑

通过真实业务接口跑多局，覆盖不同性格卡、不同产品卡、不同结束路径。

本轮关键结果：

- Bella 主批跑 9 局有效，覆盖 4 张性格卡和多类产品；9/9 最终 prompt 无未替换 `rolePlay*` 占位，9/9 未出现闭合后重问正品、价格、材质 / 清洁 / 护理服务。
- Elena 主回归 6 局，覆盖 A/B/C/D 四类性格和多种包款；6/6 最终 prompt 无占位残留，6/6 顾客回复不含“正价”，6/6 未出现心理旁白。

### 4. 结束专项

单独验证成交、再想想、不买离店这些容易卡住的结束路径。

结果：

- Bella 成交与再想想路径在修补后均能输出 `<?END_CHAT>`。
- Elena 成交和再想想路径均已跑通。

## 八、为什么不能只靠提示词

这次最重要的经验是：多轮角色扮演里，“不要重复问”“记住历史”“别倒退”这类要求，不能只靠提示词的一句话解决。

原因有三点：

1. 模型会被当前轮导购输入牵引。导购再次提到旧话题时，模型可能顺着又问一遍。
2. 长对话里，模型对历史的注意力不稳定，隔轮绕回比邻接重复更难压。
3. 状态标签本身也可能漏写或写错，尤其在顾客一句话同时包含动作、顾虑和表态时。

所以本次采用“软规则 + 硬护栏”的组合：

- prompt 负责告诉模型该怎么演、怎么说、怎么推进。
- `<?STATE>` 负责让每轮输出结构化状态。
- 后端负责聚合状态、同类归并、兜底漏记、追加已聊提醒。
- prompt trace 负责把最终入模内容暴露出来，便于定位真实问题。
- 接口批跑负责验证它在真实链路里是否有效。

这也是 COACH 场景和标准 AI 陪练搭建方式最大的不同：它不是单点 prompt 调优，而是内容工程和业务代码共同形成闭环。

## 九、后续复制新场景的方法

如果后续继续扩展 COACH 新场景，建议按下面顺序做：

1. 先定义顾客心理账户：这个顾客为什么进店、最怕什么、什么情况下会被打动。
2. 再拆场景边界：不要把正价、奥莱、私域、送礼、自用、首次购买等语境混在一起。
3. 写行为卡，不只写性格标签：每张卡都要有说话气质、最先关切、证据阈值、追问策略、被打动信号、被推开信号。
4. 建同类关切表：把可能重复的问题归并成有限类别，并定义什么叫“被导购接住”。
5. 设计状态账本：明确 `关切 / 动作 / 表态` 三字段如何记录，哪些内容必须进入历史。
6. 做对抗审查：先用最容易复发的问题打穿，再进入真实接口批跑。
7. 补代码护栏：只要发现纯 prompt 压不住的重复、漏记、倒退，就优先考虑状态聚合、兜底和 trace，而不是继续堆更多“不要”。
8. 输出两类报告：内部报告保留机制、代码和 record 证据；客户报告只讲场景、问题、优化、测试结果和真实对话。

## 十、当前边界和风险

本轮已经收口的是双场景角色扮演链，不代表全部三链都已完成。

当前边界：

- 角色扮演链已完成双场景调优、代码配合和真实接口回归。
- 实时评估链、整体评估链不属于本轮角色扮演调优范围，后续仍需单独按评估标准验证。
- 产品价格、活动、库存和门店规则仍以业务主数据和门店最新口径为准，角色扮演链不负责事实核查。
- 模型仍有随机性，后续客户验收和真实 SA 试用时，仍建议保留 prompt trace 与小范围回归机制。

优先建议的下一步：

1. 把这套角色扮演链方法固化成“新场景搭建检查清单”。
2. 评估链补齐后，重点验证事实核查、话术建议和扣分逻辑是否能承接角色扮演链不查知识库后的职责。
3. 后续新增产品池或改活动价时，至少跑一轮小回归，检查产品锚点、价格表达和结束路径。
