# Coach AI 销售陪练系统 · 内容工程仓 — AI 协作说明（canonical）

> 本文件是跨工具的单一来源（Codex / 多数工具原生读 AGENTS.md）。
> Claude Code 侧 `CLAUDE.md` 仅 `@AGENTS.md` 引入本文件，避免双份维护。
> 详细规范全文在 `docs/conventions/`，由全局 hook 在编码时自动注入，这里只放"地图 + 基线 + 指针"。
> **本仓特殊性：纯文档/prompt 工程仓，无应用代码。** 下文凡提"代码/编码"，在本仓语境等价于"文档 / prompt / 知识库内容"。

## 1. 项目地图（让 AI 快速定位）

- **一句话定位**：为 COACH（蔻驰）门店销售（SA = 门店销售，被陪练的真人学员）打造的 **AI 销售陪练系统的内容工程仓**——用纯文档 / prompt 工程产出两套跑在**阿里云百炼**平台的资产：六库 46 篇标准知识库（KB1–KB6）+ 场景化**三链**陪练 prompt（①角色扮演 ②实时评估 ③整体评估）。仓内无应用代码，产物经百炼控制台部署与实测验收。
- **载体 / 技术栈**：纯 Markdown（Obsidian vault）+ 少量 .xlsx/.docx/.png 原始素材；无编译 / 构建 / 运行时。
- **构建 / 测试 / 本地启动**：**均无**（纯文档）。质量验收靠两条仓外的人工/平台动作——① 百炼建库后真跑命中测试 T1–T21；② 作者对抗式自测 prompt 行为。仓内唯一可自动跑的是 `bash scripts/verify-local.sh`（**文档卫生检查**，见 §4）。

### 目录地图（"上游原料 → 下游两条工程线"的物料流水线）

- `辅助资料整理/` — **上游原始物料源**。全二进制素材（.xlsx/.docx/.png/.jpg）、无 .md：`企业专有名词/`（包型/颜色黑话词典）、`历史poc场景搭建/`（旧版 POC 的 SA 真人反馈 + 旧评估维度）、`最新完成陪练场景提供材料/`（场景1 Elena 画像来源）。
- `知识库搭建/` — **下游线一：知识库层**。把零售资料 / 课程提炼成 KB1–KB6 共 **46 篇**（KB1=3 / KB2=11 / KB3=18 / KB4=5 / KB5=5 / KB6=4）。架构定稿 **v2.0**；含生成管线两道 prompt、部署配置 + 21 条命中测试清单、变更追溯台账。**上线源 = 顶层 `具体KB内容/`**（`output_v2/` 不是上线源：它是上一交付批次的快照 / 中间稿，其 `交付总览.md` 自报 45 篇五库口径，成品 KB 已不在该目录）。
- `陪练场景搭建/` — **下游线二：场景 / prompt 层**。三链各发布为独立**百炼 Agent 2.0**，编排由业务系统控制、提示词下沉、变量经后端注入。当前只深耕**场景1（Elena）的"角色扮演链"**（已迭代至 v8.x，正式版 = 无版本号的 `agent最新使用提示词.md`；现阶段性暂停、待补真实销售 / 用户信息），评估两链未开工。含总体设计台账、场景1 全部产物、随机性设计（随机池已对接后端代码 `NstrRoleplayCard`）。
- `docs/`（rig 接入新增）— `conventions/`（项目规范全文，编码时自动注入）、`adr/`（跨域决策记录）、`plans/`（计划模板）。

## 2. 行为基线
1. 先想后写：歧义先澄清，不擅自假设。
2. 简洁优先、不过度设计。
3. 外科手术式改动，风格随周围内容（文档语境：不顺手重排无关章节、不改既有版本号语义）。
4. 目标驱动：成功标准可验证，能本地验证就验证到通过。

> 源自 Karpathy 四原则，与全局 `~/.claude/conventions.md` 同源；冲突以本项目为准。

## 3. 规范遵守（重要）
- 本项目的**完整规范在 `docs/conventions/`**（写作 / 命名 / 版本 / 归档 / 分库约定），会在你开始动文档时被注入上下文，**必须遵守**。
- 优先级：**本项目规范 > 全局个人偏好**，冲突以本项目为准。
- 本仓规范多为 **B 桶语义约束**（如异议写法、版本号语义、归档时机），靠注入落地；少数 **A 桶机器可判定**（KB 文件命名、百炼变量名契约、KB 篇数）由 `scripts/verify-local.sh` 的文档卫生检查兜底；机器判不了的语义在收尾 `code-reviewer` 复核。
- **受保护的定稿产物**（§1 列出的单一事实源、`.claude/protected-paths.txt` 全表）：改它们=改线上百炼资产，**动手前必须先停下问用户**，不要自行绕过 guard。

## 4. 本地自验证（L0 Harness）— 文档卫生检查
- 本项目无 compile/test/run。`bash scripts/verify-local.sh` 做**文档卫生检查**，严格度=平衡档：
  - **硬拦（不过别收工）**：KB 文件命名规范、定稿产物存在性、线上 prompt/随机池的百炼变量名契约（`roleplay_personality` / `roleplay_product`，防与后端 `NstrRoleplayCard` 脱节）。
  - **只提醒（迭代期不拦，避免误伤）**：KB 各库篇数（对齐 v2 的 3/11/18/5/5/4=46）、内部链接 / Obsidian 双链死链、KB 元信息五字段。
- 改完文档自行跑一次，硬拦项不过别说"已完成"。
- **真正的质量验收在仓外**：百炼命中测试 T1–T21（需登录百炼控制台人工跑）+ prompt 对抗式自测——脚本不伪装能验这些。

## 5. 变更流程：先判 意图×风险，再走对应路径
本仓语境：「代码」≈ prompt/KB 文档，「接口/线上契约」≈ 百炼变量名 / 随机池 JSON / 上线源篇数。

| 意图 × 风险 | 走的路径 |
|---|---|
| **QUERY**（查阅 / 排查，不改产物） | 0 流程——直接查、答。 |
| **改动 · 低**（局部润色、单篇 KB 小改、可逆） | 快路径：注入规范 → 改 → `verify-local`（文档卫生）。照常 git。 |
| **改动 · 中**（跨文档改、改一条链结构 / 新增场景 / 调随机池） | 快路径 **＋ 设计前段**：`superpowers` brainstorm（澄清边界）→ write-plan（验收写成可执行，套 `docs/plans/_template.md`）→ **你审批** → 执行。 |
| **改动 · 高**（改百炼变量名 / 随机池契约 / 知识库架构 / 动受保护定稿产物） | 中档全部 **＋ `/rig:adr` 记跨域决策 ＋ 收尾跑 `code-reviewer` 复核 ＋ 先停下问用户**（碰受保护产物 = 改线上）。 |

> **openspec 暂未启用**（本项目为改动驱动：持续迭代打磨内容、无正式需求工单）。决策回溯靠"总体设计台账 + 决策日志 + 变更追溯台账 + `归档/`"这套既有机制。若未来转为按客户正式需求清单逐条交付并需签字验收，再 `/rig:new-change` 启用 openspec。

**硬规则「改完必验证」**：凡动了产物文档，`verify-local` 硬拦项必过才算收工；纯 QUERY 除外。

**可用命令**：`/rig:adr` 记跨域决策 · `/rig:feature-spec`（本仓 N/A，无代码可扫）· `/rig:learn` 沉淀踩坑、晋升规则 · `/rig:review` 收尾语义复核。

## 6. 收尾与沉淀
- **收尾复核**：改动完成后调 `code-reviewer` 子 agent 做一次 fresh-context 对抗式语义审查（遵守度 / 偏离度 / 完成度，只报不改）——专兜文档卫生检查判不了的语义（如异议是否写成了完整台词、版本号该不该升）。
- **「为什么」唯一权威**：跨域长期决策毕业到 `docs/adr/`（用 `/rig:adr`）。本仓既有的决策日志（场景台账 §10、知识库架构 D1–D17）是事实主力；ADR 用于真正跨域、需长期回溯的大决策。
- **后向真相**：feature-spec 以"代码为准"沉淀域设计——**本仓无代码可扫，N/A**；该职能由业务架构台账（场景 / 知识库架构文档）天然承担。
- **越用越聪明**：踩了坑 → `/rig:learn` 沉淀成 lesson（落 `docs/lessons.md`，不注入）；复现多了、验证后**经你确认**晋升——机器可判的进 `scripts/verify-local.sh`（A 桶硬拦）、需判断的进 `docs/conventions/`（B 桶注入）。
