# 0002 - 陪练场景搭建目录分层骨架（L1–L5）

## 背景与问题

`陪练场景搭建/` 由"场景1 单场景打深"形成扁平堆叠：评估链草案、随机池、各类记录、归档混在场景1 目录或散落顶层。要扩到场景2/3… 时，缺一套可复制的标准结构，且公共资产（评估链、随机机制）与场景专属内容纠缠在一起。

## 决策

我们决定把 `陪练场景搭建/` 重组为 **L1–L5 分层骨架**，并把场景子目录**标准化成一个模板**：

- **L1 项目级**：`0_项目总览.md`（← 台账 `场景搭建_总体设计与进度.md` 改名）+ `0_提示词写作规范.md`（业务写作方法论全文）
- **L2 公共资产**：`公共_跨场景复用/{评估链提示词, 随机机制}/`
- **L3 场景层标准模板**：`场景N_<画像角色>/` = `agent最新使用提示词.md`（SSOT）+ `草案/ 记录/ 归档/ 画像素材/`
- **L4 知识库**：`知识库搭建/` 独立一层，本次不动
- **L5 参考**：`参考资料/` 只读素材

关键子决策：① **随机池 = 全局单一**，归 L2 `公共_跨场景复用/随机机制/`（对接后端单一 `nstr_global_config`）；② 场景目录改清爽名（`场景1_轻奢包追求者Elena_正价` → `场景1_质感自用Elena`），**SSOT 文件名保留**（`agent最新使用提示词.md`、`最新随机池内容.md` 不改名）；③ 状态用**目录**区分（草案 / 记录 / 归档），不用文件名后缀。

## 为什么

- **可复制模板**：新场景照抄 L3 模板即可，这就是要的"可扩展体系"。
- **关注点分层**：公共资产抽到 L2、跨场景复用；场景专属归 L3；只读素材归 L5。
- **随机池归公共**：它是跨场景全局配置（后端一份 config），不属于某个场景。
- **SSOT 文件名保留**：`agent最新使用提示词.md` 等被多处引用且标"唯一在用"，改名徒增同步成本（与 ADR 0001 同源的"聚焦真·线上资产"思路）。

## 备选方案

- **保持扁平**：维持现状 — 不选，无法标准化扩展、公共与专属纠缠。
- **状态用文件名后缀区分**（_草案 / _归档）：— 不选，易乱，不如目录清晰；正是用户图里要点否定的写法。
- **随机池放进场景1**（按原图位置）：— 不选，它是全局单一资产，放某场景语义不对。

## 影响与代价

- 正面：可扩展场景模板成型；公共资产复用清晰；状态一目了然。
- 代价：场景1 改名 + 文件移位后，rig 产物里所有相关路径必须同步更新（`AGENTS.md` / `docs/conventions/*` / `.claude/protected-paths.txt` / `scripts/verify-local.sh`），否则悬空 / guard 失效。
- 风险：受保护 SSOT 移动需谨慎；本次用 `git mv` 保留历史、移动后实测 `verify-local` + `guard` 仍绿。
