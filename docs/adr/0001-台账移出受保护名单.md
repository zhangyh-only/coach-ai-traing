# 0001 - 台账移出受保护名单

## 背景与问题

`陪练场景搭建/场景搭建_总体设计与进度.md`（下称"台账"）此前被列入 `.claude/protected-paths.txt` 受保护名单，与角色扮演链正式版、随机池、KB 定稿等并列——动它前必须停下问用户、且 guard 钩子硬拦不认口头授权。

但台账的本质是**高频维护的进度/决策记录**：每完成一次设计对齐、产出一份链路草案、做出一个关键取舍，就要回写 §5–§13 对应章节。把它当"定稿冻结产物"，导致每次回写都要"临时放行 → 改 → 恢复保护"，迭代期摩擦很大。本次开发实时评估链、回写台账时这个矛盾显性化。

## 决策

我们决定把台账 `场景搭建_总体设计与进度.md` **从 `.claude/protected-paths.txt` 移出**（注释该行并留说明、不静默删，保留可回溯），允许 AI 常规回写。

**保护名单的分类标准就此厘清**：只保护"误改 = 改线上百炼资产"的真定稿——角色扮演链正式版 `agent最新使用提示词.md`、随机池 `最新随机池内容.md`、角色扮演随机机制说明、KB 定稿、测试报告、调优记录等；**高频维护的设计/进度/决策记录类文档不纳入**。这些真·线上资产的保护**保持不变**。

## 为什么

- 台账不是百炼线上跑的资产，改它**不直接动线上**；它是仓内设计/决策记录，天生需要高频更新。
- "受保护 + 高频更新" = 每次放行，摩擦大、拖慢迭代，与台账"持续维护"的设计定位冲突。
- 误改防护有充分替代：台账自身**"保留旧结论 + 决策日志逐条带日期 + 变更记录"**写作约定（B 桶规范、编码时注入，每次回写都遵守）+ **git 可回溯** + **收尾 code-reviewer 复核**。
- 保护机制应聚焦"误改代价最高"的真·线上资产，名单越聚焦越不容易让人麻木地绕过。

## 备选方案

- **保留保护 + 每次回写前问用户放行**：维持现有安全级别，但每次都要人工放行一次，长期摩擦大、拖慢迭代 — 不选。
- **保留保护 + 会话级一次性放行**：每个会话开头授权一次 — 仍需每会话操作、半吊子，且 guard 钩子无"会话级豁免"机制 — 不选。

## 影响与代价

- 正面：台账可被 AI 常规回写，三链开发/决策记录的迭代不再被保护机制打断。
- 代价：台账失去 guard 硬拦，误改风险**略升**；改由软约定（写作规范）+ git + 收尾复核兜底。
- 风险：若 AI 误删/误覆盖台账内容 → git 可回退；`verify-local` 的相对链接死链提醒可发现部分断链；收尾 `code-reviewer` 复核语义。若未来台账误改频发，再考虑加轻量校验或恢复保护。

## 后续

- `.claude/protected-paths.txt` 已注释台账行并留说明指向本 ADR。✅
- 台账 §10 决策日志、§13 变更记录已各记一条本次变更。✅
- 后续新增"高频设计/记录类"文档时，按本 ADR 的分类标准判断：默认不纳入保护名单，只保护真·线上资产。
