# 0006 - 场景2随机池架构：性格池走独立 config_key、草案阶段不灌 SSOT

> **修订状态（2026-07-02）**：本 ADR 的“草案阶段不灌 SSOT”节奏已被 ADR 0007 取代。当前场景2提示词和随机池均已提升为在用资产，并归入场景2目录。保留本文作为历史决策记录。

## 背景与问题

场景2（奥莱私域轻奢尝鲜族 Bella）需要自己的 4 张性价比型性格卡。但现有随机池 SSOT（`陪练场景搭建/场景1_质感自用Elena/随机机制/最新随机池内容.md`）有两个约束：

1. **它是场景1 专属的**——文件标题即「场景一 · 角色性格 / 产品 随机池」，4 张性格卡 content 带质感自用画像色彩（如 D"配不配自己"、A"耐不耐用值不值"）。
2. **config_key 全局单一、无场景路由**——性格池 `nstr.roleplay.personality-pool`、产品池 `nstr.roleplay.product-pool` 各一个 key，后端注入时不区分场景。

直接把 Bella 4 卡塞进同一个 personality-pool，会与 Elena 卡混抽、场景1/2 无法区分。随机池又是受保护定稿产物（改它＝改线上百炼资产）。故需先定"性格池如何按场景隔离"的架构，以及"草案阶段要不要现在就动 SSOT"的节奏。

## 决策

我们决定：**场景2 性格池采用独立 config_key 隔离（方案A）作为目标架构；当前草案阶段不灌入受保护的随机池 SSOT**——Bella 4 卡先以草案形式留在 `场景2_奥莱私域轻奢Bella/随机机制/最新随机池内容.md`，待场景2 提示词稳定、要正式上线时，连同后端"按场景选 config_key"的路由改造一起灌入。

- **目标架构（方案A）**：新增场景2 专属性格池 key（如 `nstr.roleplay.personality-pool.scene2`），后端按场景 ID 选性格池 key 注入；**产品池复用场景1 同一 key**（场景2 产品池＝Tabby/Brooklyn，与场景1 一致，见设定 spec §4）。
- **边界**：本 ADR 只定性格池隔离方式与灌入节奏；后端"按场景选 key"的具体实现属业务后端，不在本仓。

## 为什么

- **草案阶段不碰线上资产**：场景2 提示词、4 卡都还是草案（待补真实信息、待百炼实测）；此时改受保护的随机池 SSOT + 改 guard，风险大、收益低。
- **现在灌也跑不起来**：方案 A/B 的"按场景路由"都依赖后端改造；后端没好之前，即便灌入 SSOT，场景2 仍会与场景1 混抽、无法真跑——"现在灌"几乎没有实际收益。
- **方案A 隔离最干净、改动最集中**：独立 config_key 让 Elena/Bella 卡物理分离、互不污染；落地只需"加一个池 + 后端按场景选 key"，不动 `NstrRoleplayCard` 数据结构。
- **与"先草案就绪、上线时统一灌入"的节奏对齐**：内容先做扎实，架构改造留到场景2 整体上线时一次性做，避免半成品动线上。

## 备选方案

- **方案B（`NstrRoleplayCard` 加 scene 字段、一池混装）**：不选——改后端数据结构最重、混装易错，且同样依赖后端改造才能按 scene 过滤。
- **现在就按方案A 灌入 SSOT**：不选——动了受保护产物 + 改 guard，但后端路由未就绪，场景2 仍跑不起来，收益有限却已担风险。
- **场景2 复用场景1 同一性格池**：不选——Elena 卡带质感自用色彩、Bella 是性价比/怕踩坑，混抽会让两个场景的顾客性格穿帮。

## 影响与代价

- 正面：草案阶段零线上风险；目标架构清晰（独立 key）；产品池复用降低重复；场景隔离干净。
- 代价：① 场景2 **暂不能在百炼真跑**（性格卡未进池、后端路由未做）——这是有意的节奏选择；② 将来上线时需**后端配合**：加场景2 性格池 key +「按场景选 key」路由 + 灌 `nstr_global_config`（后端动作，AI 改不了）；③ 灌入时随机池 SSOT 为受保护改动，需走"开关代改→改完恢复 guard + 收尾 code-reviewer"。
- 风险：随机池"全局单一"的假设散落在文档/机制说明里（SSOT 标题、verify-local 契约检查等）；引入多场景 key 时需同步梳理，避免遗漏。缓解：上线灌入时以本 ADR 为索引逐处核对。

## 后续
