# 0007 - 随机机制归属各场景目录

## 背景与问题

ADR 0002 曾把随机机制放在 `公共_跨场景复用/随机机制/`，因为当时只有场景1 Elena，且后端随机池配置被理解为全局单一。随着场景2 Bella 完成提示词和随机池调优，这个结构开始暴露问题：

1. 场景1 Elena 与场景2 Bella 的顾客心理账户不同，性格卡不能混池。
2. 公共随机池文件夹会让“场景专属设置”和“跨场景公共资产”混在一起。
3. 场景2 的最新提示词和 Bella 性格卡已经在运行侧使用，继续放在 `草案/` 或公共随机机制下会误导后续维护。
4. 用户明确要求：公共随机池文件夹不需要了，随机设置和对应随机池内容放回各场景下。

## 决策

我们决定：**随机机制不再作为 L2 公共资产维护，而是归属到各场景目录。**

新的场景目录标准为：

```text
场景N_<画像角色>/
├── 最新使用提示词.md
├── 随机机制/
│   └── 最新随机池内容.md
├── 草案/
├── 记录/
├── 归档/
└── 画像素材/
```

当前落地：

- 场景1 Elena 随机池：`陪练场景搭建/场景1_质感自用Elena/随机机制/最新随机池内容.md`
- 场景1 随机机制说明：`陪练场景搭建/场景1_质感自用Elena/随机机制/角色扮演随机机制说明.md`
- 场景2 Bella 随机池：`陪练场景搭建/场景2_奥莱私域轻奢Bella/随机机制/最新随机池内容.md`
- `公共_跨场景复用/随机机制/` 删除，公共层只保留真正跨场景复用的评估链、报告和辅助说明。

## 为什么

- **场景边界更清晰**：Elena 是正价 Retail 质感自用顾客，Bella 是奥莱 Outlet 轻奢尝鲜顾客；两者的性格卡、关切分类和推进阈值不同，随机池放在场景内更符合语义。
- **避免混抽风险**：随机池跟场景放在一起，能在内容层直接看出“这个池服务哪个场景”，减少把 Bella 顾虑灌回 Elena 或把 Elena 价值逻辑套到 Bella 的维护风险。
- **结构更可复制**：新增场景时直接复制 L3 模板，补 `最新使用提示词.md` 和 `随机机制/最新随机池内容.md` 即可，不需要再判断哪些随机内容属于公共、哪些属于场景。
- **对齐当前运行事实**：用户已确认各场景最新提示词和随机池内容都在使用，目录结构必须反映“在用资产”而不是“草案 / 公共临时层”。

## 备选方案

- **继续保留公共随机池目录**：不选。它会让场景专属性格卡看起来像跨场景公共资产，后续维护容易误用。
- **公共层只放产品池，性格池放场景层**：不选。虽然当前两个场景产品池内容一致，但用户要求“自己的设置和对应的随机池内容”放到各场景下；保留一份公共产品池仍会让单场景目录不自洽。
- **只在文档说明中声明场景隔离，不移动文件**：不选。目录本身仍会误导，且 `verify-local`、guard、AGENTS 会继续指向旧公共层。

## 影响与代价

- 正面：场景资产闭环更清楚；公共目录更轻；后续新增场景更容易复制；保护路径和本地验证能直接检查每个场景的在用资产。
- 代价：产品池当前在场景1、场景2各保留一份相同内容，后续若产品池统一调整，需要同步更新两个场景文件，或另行设计产品池共享机制。
- 风险：历史 ADR / plan / 测试记录中仍可能保留旧公共随机池路径。当前做法是更新现行规范、AGENTS、verify、guard 和关键记录路径；历史语境中若保留旧表述，以本 ADR 和现行目录为准。

## 后续

- 将场景2 `草案/角色扮演链提示词_Bella_v0.4_草案.md` 提升为 `最新使用提示词.md`，将 `草案/随机池_Bella性格卡_草案.md` 提升为 `随机机制/最新随机池内容.md`。
- 后续新增场景时，不再创建公共随机池文件夹；直接在场景目录下创建 `随机机制/最新随机池内容.md`。
