# 后端「状态锚点」机制设计方案（草案）

> **状态**：草案，待用户审 + 后端评估可行性。
> **定位**：根治"隔轮绕回同一关切 + 结尾倒退"——这类纯提示词压不死的状态连续性问题（ADR 0005 风险① 的兜底）。
> **归属**：主体是**后端业务代码**改动（AI 改不了）；内容仓侧只配合改 1 句提示词 + 提供提取规则。
> **日期**：2026-06-26 ｜ 相关：[ADR 0005 直调模型](../adr/0005-角色扮演链切回直调模型.md)、[接口测试记录](../../陪练场景搭建/场景1_质感自用Elena/记录/接口测试输出/)

## 一、背景：为什么纯提示词不够（实测证据）

ADR 0005 把角色扮演链改为「后端组装 `[系统提示词 + 全量历史]` 直调 qwen3.7-plus」，并**预判风险①：直调全靠提示词认历史、无 Agent 记忆兜底**。2026-06-26 用业务接口（baseId=497）实测 2 局，证实该风险：

| 局 | 性子 | 现象 |
|---|---|---|
| rec157386 | B 比价型 | 在"会不会变旧/娇气/染色/磨损"**一个大主题换皮纠缠 6 轮**，第 6 轮绕回已答过的"牛仔裤染色" |
| rec157387 | A 理性型（提示词已加"看底层顾虑+自检"修复后） | **邻接换皮改善**（留印答完转五金），但"光面留印/划痕"**隔轮绕回 3 次**（第 1/3/6 轮）；结尾"那就这只吧"后**倒退**"还没决定、重做摸皮看做工" |

**结论**：
- "看底层顾虑、答过翻篇"对**邻接换皮**有效；导购**点破**（"这我刚说过了"）能让 Elena 当轮收口。
- 但**隔轮绕回**（过两轮又回老关切）和**结尾倒退**（已表态又反悔），纯提示词压不死——认历史靠模型自觉，长对话兜不住。与 v8.x 历史结论一致。
- 这类问题的关注点应从"薄提示词"落到"后端每轮注入状态"这个更对的载体。

## 二、机制设计

### 核心
后端每轮组装喂模型的消息序列时，**根据这一局已发生的对话动态算出一段「本局状态锚点」**，作为**一条 system 消息追加在序列最末**（最新一轮 SA/user 消息之后）。让模型生成 Elena 下一句前，最后看到的就是"这局已经发生了什么"，强制认历史、防绕回与倒退。

### 锚点内容（动态生成，示意格式）
```
[本局状态 · 仅供你认历史，绝不复述给导购]
· 已问过、导购也答过的（别再换说法重提）：光面耐刮/留印、五金掉漆、容量够不够
· 你已经做过的（别再申请第二遍）：试背过、调过肩带、掂过分量、打开看过内里
· 当前进度：你已说过"就这只吧"——别再退回"还没决定"或重做摸皮看做工
```
> 注意：锚点喂给模型的是「**已经聊过 X**」（事后事实），**不是**「该聊 X」（事前清单）——所以它**不会把对话固定化、不压平性格**，只挡重复和倒退。这与"提示词只写机制不写内容剧本"的原则一致。

### 三类信息怎么算（后端规则，零额外 LLM 成本）
1. **已聊关切清单**：扫历史 Elena 文本，用「关键词 → 关切大类」映射提取（耐用/留印/划痕、容量、五金、保养、价格/值不值、颜色…），按**大类**去重。粒度落在"大类"是关键——皮面留印和五金磨花都归"耐用/会不会变旧"，才挡得住换部位绕回。
2. **已做动作**：扫历史 Elena **旁白（括号内）** 关键词：试背/背上肩、打开/翻看内里、掂量/拿起、调肩带…
3. **当前进度/倾向**：检测历史 Elena 是否出现购买倾向句（"那就这只""就它了""帮我包/开单"），出现则标"已表态买 → 防倒退"。

> 提取规则可**直接移植** `scripts/build_report.py` 里的 `CONCERN_KW` 关切映射 + 旁白动作/倾向识别（已在测试里验证可用）。

### 放置与格式
- **推荐**：独立 `system` 消息插在消息序列最末（不污染 SA 台词、提醒最靠近生成点）。
- 备选：拼进最新 user 消息尾部加 `[本局状态…]` 标记（实现更简，但混入 SA 发言里）。

### 提示词侧配合（内容仓侧，我改，仅 1 句）
系统提示词加一句（放铁律1 或开头）：
> "对话里可能出现 `[本局状态]` 这样的提醒，那是给你认历史用的——照它别重提、别倒退，但**绝不把它读出来或复述给导购**。"

### 与直调架构的衔接（增量极小）
- 现状：后端组装 `[系统提示词 + 全量历史]` → 直调。
- 增量：组装时**多一步**——遍历全量历史算出状态锚点，作为一条 system 消息拼在末尾。其余不变。纯规则、无额外模型调用、无明显延迟。

## 三、验收（实现后我用接口验）
- 复跑 B 比价 / A 理性场景（同这次的施压节奏），看：① 同一大类关切答过后**不再隔轮绕回**；② 已表态买后**不倒退**、不重做开场动作；③ 锚点本身**不被复述**出来。
- `build_report.py` 出修复前后对照 HTML。

## 四、决策点（待用户/后端定）
1. 锚点放 **system 消息**（推荐）还是拼 user 尾部？
2. 关切提取用**规则**（推荐起步、零成本）还是轻量 LLM 摘要（更准但加一次调用）？
3. 要不要再加**事后重复检测**（模型输出后规则查重、命中则重生成）？建议先只做事前锚点，不够再加。

## 五、分工
- **后端**（业务代码，AI 改不了）：组装逻辑加"算锚点 + 插 system 消息"；移植 `build_report.py` 的提取规则。
- **内容仓（我）**：① 提示词加上面那 1 句配合；② 把提取规则整理成后端可直接用的清单；③ 实现后用接口验收 + 出对照 HTML。
- **决策记录**：方案定了补 ADR 0006（角色扮演链加后端状态锚点）。
